はじめに：責任共有モデルとこの章の範囲

SAA-C03試験のDomain 1「Design Secure Architectures」は、出題全体の30%を占める最大の比重を持つ領域です。 公式試験ガイドでは、この領域は次の3つのタスクステートメントに分かれています。

  • Task 1.1: AWSリソースへの安全なアクセスの設計（IAM、フェデレーション、複数アカウント戦略など）
  • Task 1.2: セキュアなワークロード/アプリケーションの設計（VPCセキュリティ、脅威対策サービスなど）
  • Task 1.3: 適切なデータセキュリティ管理策の決定（暗号化、鍵管理、データ保護ポリシーなど）

この章の土台になるのが「責任共有モデル（Shared Responsibility Model）」です。AWSは「クラウドそのもののセキュリティ （Security of the Cloud）」、つまりデータセンターの物理セキュリティ、ハードウェア、基盤ソフトウェア、 仮想化基盤、ネットワークインフラの保護に責任を持ちます。一方、利用者は「クラウド内のセキュリティ （Security in the Cloud）」、つまりOSのパッチ適用、ネットワーク・ファイアウォールの設定、 IDとアクセス管理、データの暗号化、クライアントサイドの保護に責任を持ちます。 どこまでをAWSが管理し、どこからを利用者が管理するかは、EC2のようなIaaS的サービスか、 RDSやLambdaのようなマネージド度の高いサービスかによって境界線が変わります。 マネージド度が上がるほど、AWSが管理する範囲は広がり、利用者の責任範囲は狭くなります。

学習のポイント：試験では「これは誰の責任か」という視点の設問が頻出します。 たとえば「ゲストOSのパッチ適用」「セキュリティグループの設定」「保管データの暗号化キーの管理」は利用者側、 「基盤ネットワークインフラの物理的保護」「ホストOSと仮想化レイヤーの管理」はAWS側、という切り分けを押さえておきましょう。

1.1　AWSリソースへの安全なアクセス設計

IAMの基本要素

IAM（AWS Identity and Access Management）は、「誰が（プリンシパル）」「何に（リソース）」「どんな操作を」 できるかを制御する、AWSのアクセス制御の中核サービスです。中心となる要素は次の4つです。

  • IAMユーザー：人やアプリケーションに対応する、長期的な認証情報を持つアイデンティティ
  • IAMグループ：ユーザーの集合。グループにポリシーをアタッチして権限をまとめて管理する
  • IAMロール：特定のプリンシパル（IAMユーザー、AWSサービス、外部IDプロバイダーなど）が 一時的に引き受ける（Assumeする）ためのアイデンティティ。長期的な認証情報を持たない
  • IAMポリシー：JSON形式で許可・拒否するアクションを定義するドキュメント

実務やアーキテクチャ設計では「長期的なアクセスキーを持つIAMユーザーを増やさず、 必要なときだけロールを引き受けて一時的な認証情報を使う」ことが推奨されるベストプラクティスです。

ポリシー評価ロジック

複数のポリシーが同じリクエストに関与する場合、AWSは次の優先順位で評価します。

  1. 明示的な拒否（Explicit Deny）：いずれかのポリシーに明示的なDenyがあれば、 他のポリシーが許可していても最終結果は拒否になる（最優先）
  2. 明示的な許可（Explicit Allow）：明示的な拒否がなく、いずれかのポリシーに 明示的なAllowがあれば許可される
  3. 暗黙の拒否（Implicit Deny）：明示的な許可も拒否もなければ、デフォルトで拒否される

ポリシータイプの組み合わせ方によって、評価が「OR条件」になる場合と「AND条件（交差）」になる場合があります。

組み合わせ評価の考え方
IDベースポリシー ＋ リソースベースポリシー どちらか一方で許可されていれば許可（OR）。ただしどちらかに明示的拒否があれば無効
IDベースポリシー ＋ アクセス許可の境界（Permissions Boundary） 両方の許可が重なる範囲のみ有効（AND）。境界は権限を付与せず、上限を定めるだけ
IDベースポリシー ＋ SCP／RCP ＋ リソースベースポリシー 関与するすべてのポリシータイプで許可されて初めて許可（AND）

覚え方：SCPやアクセス許可の境界は「権限を与えるもの」ではなく「権限の上限を絞るガードレール」です。 IAMポリシーで許可されていても、SCPで許可されていなければ最終的にはアクセスできません。

多要素認証（MFA）

MFAは、パスワードなどの通常のサインイン認証情報に加えて、追加の認証要素を要求する仕組みです。 2026年時点のAWS公式ドキュメントでは、次の3種類のMFAがサポートされています。

  • パスキー／セキュリティキー：FIDO標準に基づく方式。フィッシング耐性が最も高く、AWSが第一に推奨
  • 仮想認証アプリ（仮想MFAデバイス）：TOTPアルゴリズムを使うスマートフォンアプリなど
  • ハードウェアTOTPトークン：専用ハードウェアデバイスが数字コードを生成する方式

なお、SMSテキストメッセージによるMFAはAWSではサポートが終了しています。 ルートユーザーについては、すべてのアカウントタイプ（単独アカウント、Organizationsの管理アカウント、 メンバーアカウント）でMFAが必須（enforced）とされています。ルートユーザーとIAMユーザーはそれぞれ 最大8台までのMFAデバイスを登録でき、複数登録しておくことで一部のデバイスを紛失してもアクセスを継続できます。

IAM Identity Center によるフェデレーションアクセス

IAM Identity Center（旧AWS SSO）は、複数のAWSアカウントやAWS管理アプリケーションへのアクセスを 一元管理するためのサービスです。既存の外部IDプロバイダー（SAML 2.0対応のIdPなど）と接続してユーザー・グループを 同期することも、IAM Identity Center内で直接ユーザーを作成・管理することもできます。 組織全体で複数アカウントへの権限を1か所で管理したい場合は、AWS Organizationsの管理アカウントに展開する 「組織インスタンス（organization instance）」を使うのがベストプラクティスとされ、 複数アカウントへのアクセス権限管理はこの組織インスタンスでのみサポートされます。 利用者はアクセスポータル経由で、割り当てられた複数のAWSアカウントや権限セット（Permission Set）に シングルサインオンでアクセスできます。

「IAMのみで複数アカウントを都度フェデレーションする方式」と比べて、IAM Identity Centerは IdPとの信頼関係を1か所にまとめられる（One point of federation）ため、証明書管理やSSO設定の運用負荷を下げられる点が アーキテクチャ上のメリットです。

AWS STS・ロールスイッチ・クロスアカウントアクセス

AWS STS（Security Token Service）は、IAMロールを引き受けた際などに一時的なセキュリティ認証情報を発行する サービスです。AssumeRole API やマネジメントコンソールでのロールスイッチを使う場合、 セッション時間は次の範囲になります。

方式最小最大デフォルト
AssumeRole（CLI/API）／コンソールでのロールスイッチ 15分 ロール側の「最大セッション期間」設定値（1〜12時間の範囲で設定可能） 1時間

あるロールから取得した一時的認証情報を使って、さらに別のロールを引き受けることを「ロールチェイニング （role chaining）」と呼びます。ロールチェイニングを行う場合、セッション時間は最大1時間に制限される点に注意が必要です （EC2インスタンスプロファイルなど、ユーザー認証情報からの最初のロール引き受けにはこの制限はかかりません）。

クロスアカウントアクセスを設計する場合は、リソースを持つアカウント側でIAMロールを作成し、 アクセス元アカウントを信頼できるプリンシパルとして信頼ポリシー（trust policy）に指定し、 アクセス元はAssumeRoleでそのロールを引き受ける、という構成が基本パターンです。

複数アカウント戦略：AWS Organizations と SCP

AWS Organizationsは複数のAWSアカウントを一元管理する仕組みで、組織単位（OU）による階層管理や、 アカウント作成の一括請求（統合請求）、サービスコントロールポリシー（SCP）の適用などができます。

SCPを理解するうえで最も重要な点は、SCPはどのプリンシパルにも権限を「付与」しないということです。 SCPは、組織内のIAMユーザー・ロール（メンバーアカウントのルートユーザーを含む）が利用できる 「権限の上限（ガードレール）」を定義するだけであり、実際にアクションを許可するには、 IDベースポリシーやリソースベースポリシーで別途権限を付与する必要があります。 また、次の点も試験で問われやすいポイントです。

  • 組織のルートやOUには、デフォルトですべての操作を許可するFullAWSAccessポリシーがアタッチされている
  • SCPは管理アカウント（management account）には影響しない。効果があるのはメンバーアカウントのみ
  • あるアカウントの実効権限は、そのアカウントの上位にあるすべての階層（組織全体・OU・アカウント）で 許可されている範囲の「論理積（AND）」になる
  • サービスにリンクされたロール（service-linked role）はSCPの制限を受けない

AWS Control Towerは、Organizationsをベースに、マルチアカウント環境（ランディングゾーン）の セットアップと、SCPやAWS Configルールを使った「ガードレール」の適用を自動化・簡素化するサービスです。 新規にマルチアカウント統治の仕組みを構築する場合、Organizationsを手動で設定するよりも Control Towerを使う方がベストプラクティスに沿った初期設定を素早く整えられます。

リソースポリシー

IAMポリシーには大きく分けて、IAMユーザーやロールにアタッチする「IDベースポリシー」と、 S3バケットポリシーやKMSキーポリシーのようにリソース側にアタッチする「リソースベースポリシー」があります。 リソースベースポリシーは、ポリシーのPrincipal要素でアクセスを許可する相手を指定できるため、 自分のアカウント内のプリンシパルだけでなく、他のAWSアカウントのプリンシパルに対して クロスアカウントでリソースへのアクセスを許可する場合にも使われます。 IDベースポリシーとリソースベースポリシーのどちらでアクセス制御を行うべきかは、 「アクセス制御の管理をリソース側に寄せたいか、ID側に寄せたいか」という設計判断になります。

1.2　セキュアなワークロード/アプリケーションの設計

VPCの基本と境界防御

VPC（Virtual Private Cloud）は、AWSクラウド内に自分専用の論理的に分離された仮想ネットワークを作る仕組みです。 VPC内はサブネットに分割し、Webサーバーはインターネットからアクセスできる「パブリックサブネット」、 アプリケーションサーバーやデータベースはインターネットから直接アクセスできない「プライベートサブネット」に配置する、 という階層化（多層アーキテクチャ）が基本パターンです。パブリックサブネットかプライベートサブネットかは、 そのサブネットに対応するルートテーブルにインターネットゲートウェイへのルート（0.0.0.0/0 → igw-xxxx）が 含まれているかどうかで決まります。

プライベートサブネットのインスタンスがOSアップデートの取得などでインターネットへアウトバウンド通信をしたい場合は、 NATゲートウェイを使います。NATゲートウェイはパブリックサブネットに配置し、プライベートサブネットのインスタンスからの アウトバウンド接続を中継しますが、インターネット側からプライベートサブネットのインスタンスへ接続を開始することはできません。

セキュリティグループとネットワークACLの違い

VPC内のトラフィック制御には、セキュリティグループとネットワークACL（NACL）という2種類のファイアウォール機構があります。 AWS公式ドキュメントでは、次のように整理されています。

特性セキュリティグループネットワークACL
適用レベルインスタンスレベルサブネットレベル
ルールの種類許可（Allow）ルールのみ許可・拒否（Allow/Deny）ルールの両方
ルール評価すべてのルールを評価してから許可を判断番号の昇順でルールを評価し、最初に一致したもので判断
戻りのトラフィック自動的に許可される（ステートフル）明示的に許可する必要がある（ステートレス）

セキュリティグループはステートフルであるため、インバウンドを許可すればそのレスポンス（戻り）のアウトバウンドは 自動的に許可されます。一方ネットワークACLはステートレスなので、インバウンドを許可する場合は、対応するアウトバウンドの 戻りトラフィックも明示的に許可しておく必要があります。設計上は、セキュリティグループをアクセス制御の主要な手段とし、 ネットワークACLはサブネット単位の大まかなガードレールや、特定の送信元を拒否する補助的な制御として使うのが基本方針です。

脅威検知・データ保護サービス

Domain 1.2では、ネットワーク層の設計だけでなく、脅威を検知・防御するマネージドサービスの使い分けも問われます。

  • Amazon GuardDuty：CloudTrailの管理イベント、VPCフローログ、DNSログなどを継続的に分析し、 機械学習と脅威インテリジェンス（既知の悪性IP・ドメインのリストなど）を使って、認証情報の侵害や暗号通貨マイニング、 マルウェアなどの脅威を検知するサービス。有効化するだけで基盤データソースの取り込みが自動的に始まる
  • Amazon Macie：機械学習とパターンマッチングを使って、S3バケット内の個人情報（PII）などの 機密データを自動的に検出・分類し、パブリックアクセス設定などバケットのセキュリティ状態も評価するサービス
  • AWS WAF：CloudFront、ALB、API Gatewayなどに対するHTTP/HTTPSリクエストを監視し、 IPアドレス、リクエスト内容の文字列パターン（SQLインジェクションやXSSの兆候を含む）、リクエスト数（レートベースルール） などの条件でアクセスを許可・拒否・カウントするWebアプリケーションファイアウォール
  • AWS Shield：DDoS攻撃対策サービス。Shield Standardは追加料金なしで全利用者に自動付帯され、 Shield Advancedは有料オプションでEC2・ELB・CloudFront・Route53・Global Acceleratorに対するレイヤー3/4/7の 高度な保護と、Shield Response Team（SRT）による専門支援を提供する
  • Amazon Cognito：Webやモバイルアプリ向けのID基盤。ユーザーの認証・管理を行う 「ユーザープール」と、認証済み（または匿名の）ユーザーにAWSリソースへの一時的なアクセス権限を発行する 「IDプール」の2つのコンポーネントからなり、それぞれ独立して、または組み合わせて利用できる
  • AWS Secrets Manager：データベース認証情報やAPIキーなどのシークレットをコード内にハードコードせず、 実行時に動的取得できるようにするサービス。自動ローテーションのスケジュールを設定でき、 アプリケーション側を変更せずに認証情報を定期的に更新できる
  • AWS Security Hub（現在の正式名称はAWS Security Hub CSPM）：GuardDuty、Macie、Inspectorなど 複数のAWSセキュリティサービスや対応サードパーティ製品からの検出結果を、標準形式（AWS Security Finding Format）に 正規化して集約するサービス。AWS Foundational Security Best PracticesやCISベンチマークなどの セキュリティ標準に対する継続的なチェックも行い、単一のダッシュボードでアカウント横断のセキュリティ状態を把握できる

外部ネットワーク接続の保護

オンプレミス環境とAWS VPCを安全に接続する代表的な手段には、次の2つがあります。

  • AWS Site-to-Site VPN：インターネット経由でIPsecによる暗号化トンネルを確立し、 オンプレミスとVPCを接続する方式。比較的短期間で構築でき、帯域はインターネット回線品質に依存する
  • AWS Direct Connect：AWSとオンプレミス間を専用線で接続するサービス。 インターネットを経由しないため、通信品質が安定し、大容量データ転送に向くが、 構築のリードタイムはSite-to-Site VPNより長くなる傾向がある

可用性を重視する設計では、Direct ConnectとSite-to-Site VPNを組み合わせ、 Direct Connect回線に障害が起きた場合にVPN経由の接続へフェイルオーバーする構成もよく使われます。

1.3　データセキュリティ管理策の決定

AWS KMSによる保管時暗号化

AWS KMS（Key Management Service）は、暗号化キー（KMSキー）の作成・管理を行うマネージドサービスです。 KMSキーはFIPS 140-3レベル3で検証されたハードウェアセキュリティモジュール（HSM）で保護され、 暗号化されていない状態でAWS KMSの外に出ることはありません。KMSキーには次の3種類があります。

  • カスタマーマネージドキー：利用者が作成・管理し、キーポリシーやローテーション設定を 自分でコントロールできるキー
  • AWSマネージドキー（例: aws/s3）：AWSサービスが自動的に作成し、 そのサービスのために管理するキー。利用者はキーポリシーを直接編集できない
  • AWS所有キー：AWSサービス内部で使われ、利用者からは見えないキー

KMSキーへのアクセス制御は「キーポリシー」というリソースベースポリシーで行います。KMSキーは既定では 作成したアカウント以外からアクセスできず、キーポリシーで明示的に許可したプリンシパルのみが 暗号化・復号などの操作を行えます。

実際にデータを暗号化する際は、大きなデータを直接KMSキーで暗号化するのではなく、KMSキーで 「データキー」を生成・暗号化し、そのデータキーで実データを暗号化する「エンベロープ暗号化」という方式が使われます。 これにより、大量データの暗号化処理をKMSの外（呼び出し元）で高速に行いながら、データキー自体はKMSキーで 保護するという二層構造を実現しています。

キーローテーション

キーの自動ローテーションの扱いは、キーの種類によって異なります。

キーの種類自動ローテーション周期
カスタマーマネージドキー（対称暗号化キー） 任意（有効化するかは利用者が選択） デフォルト365日。日数はカスタマイズ可能
AWSマネージドキー 常に自動（無効化不可） 約365日ごと
非対称キー／HMACキー／カスタムキーストア内のキー 自動・オンデマンドともに非対応（手動ローテーションのみ） —

ローテーションはキーマテリアル（暗号材料）だけを更新し、キーID自体は変わらないため、 アプリケーション側のコード変更は不要です。復号時は暗号化に使われた世代のキーマテリアルが自動的に選択されます。

S3のデフォルト暗号化

Amazon S3は2023年1月5日以降、すべてのバケットと新規アップロードオブジェクトに対して、 追加コストなしでサーバーサイド暗号化（SSE-S3）をデフォルトで適用しています。 より高度な鍵管理（キーポリシーによるアクセス制御やローテーションの管理）が必要な場合は、 SSE-KMS（AWS KMSキーを使うサーバーサイド暗号化）や、二重に暗号化するDSSE-KMSを選択できます。 なお、利用者が用意した暗号化キーを使うSSE-C（Server-Side Encryption with Customer-provided keys）は、 デフォルト暗号化の設定オプションとしてはサポートされていません。

SSE-KMSを使う場合、S3からのリクエストはAWS KMSのリクエスト数クォータ（RPS）の対象になるため、 大量アクセスが見込まれるバケットでは「S3 Bucket Key」を有効化してKMSへのリクエスト回数を減らし、 コストを抑える設計もあわせて検討します。

ACMによる転送時暗号化（TLS）

AWS Certificate Manager（ACM）は、AWSのWebサイトやアプリケーションを保護するSSL/TLS証明書の発行・保管・更新を 代行するサービスです。ACMで発行した証明書自体には追加費用はかからず、実行に使うAWSリソース（ELBやCloudFrontなど）の 費用のみが発生します。

ACM証明書は原則としてリージョンリソースであり、複数リージョンのELBで同じドメイン名の証明書を使いたい場合は リージョンごとに証明書を発行（検証）する必要があります。CloudFrontで証明書を使う場合は、 バージニア北部（us-east-1）リージョンで証明書を発行しなければならない点も試験で問われやすいポイントです。

ACMが発行した証明書は、DNS検証を使っていて、かつELBやCloudFrontなど対応するAWSサービスに関連付けられている場合、 自動更新（マネージド更新）の対象になります。一方、外部からインポートした証明書や、 AWS Private CAのAPIで直接発行した一部の証明書は自動更新の対象外で、有効期限管理を利用者自身が行う必要があります。

データのバックアップ・レプリケーションとライフサイクル管理

データ保護の設計では、暗号化に加えて「データを失わない・不正に変更させない」ための仕組みも重要です。 代表的な手段として、S3のバージョニング（同一オブジェクトの複数バージョンを保持し、誤削除や上書きから復旧できるようにする機能）、 クロスリージョンレプリケーション（別リージョンのバケットへオブジェクトを自動複製する機能）、 そしてS3ライフサイクルルール（一定期間経過後に低頻度アクセスクラスへ移行、あるいは完全に削除するルールを 自動化する機能）があります。これらはデータの保持期間・アクセス頻度・コストのバランスを取りながら、 コンプライアンス要件に沿ったデータのライフサイクルを実現するために使われます。 なお、複数AZ・複数リージョンにまたがる可用性設計やAWS Backup、RPO/RTOに基づくDR戦略は、 第2章「弾力性に優れたアーキテクチャの設計」で詳しく扱います。

章末問題（5問）

各設問は4択の単一選択式です。解答を考えてから、折りたたみ内の解説を確認してください。

問1

ある企業は、複数のAWSアカウントを持つ組織で、開発アカウントのすべてのIAMユーザー・ロールに対して Amazon EC2の起動を禁止したいと考えています。IAM管理者がIDベースポリシーで個別に制御する方法は 運用負荷が高いため、組織レベルで一括して制限したいと考えています。この要件を満たす最も適切な方法はどれですか。

  1. すべてのIAMユーザーのアクセス許可の境界（Permissions Boundary）を個別に設定する
  2. 開発アカウントが所属するOUにEC2の起動を拒否するサービスコントロールポリシー（SCP）をアタッチする
  3. 管理アカウントのルートユーザーの権限を制限する
  4. 開発アカウントのVPCセキュリティグループでEC2へのアクセスを拒否する

解答: B

SCPは組織単位（OU）やアカウントに対して権限の上限（ガードレール）を設定でき、対象アカウント配下の 全IAMユーザー・ロールに一律で適用できます。個別のユーザーごとに境界を設定するA案は運用負荷が高く、 SCPは管理アカウントには適用されないためC案は要件を満たしません。セキュリティグループはネットワーク層の制御であり、 IAMアクションレベルでのEC2起動禁止には使えません。

問2

モバイルアプリのバックエンドで、ユーザーのサインアップ・サインインを管理しつつ、 認証済みユーザーに対してS3バケットへの直接アクセス用に一時的なAWS認証情報を発行したいと考えています。 最も適切なAWSサービスの組み合わせはどれですか。

  1. Amazon Cognitoユーザープールのみ
  2. Amazon Cognitoユーザープールとアイデンティティプールの組み合わせ
  3. AWS IAM Identity Centerのみ
  4. AWS Secrets Managerのみ

解答: B

ユーザープールはユーザーの認証（サインアップ・サインイン）を担当し、JWTトークンを発行します。 このトークンをアイデンティティプールに渡すことで、AWS STSから一時的なAWS認証情報を取得し、 S3などのAWSリソースに直接アクセスできるようになります。ユーザープール単体ではAWS認証情報は発行されません。 IAM Identity Centerは主に社内の従業員（ワークフォース）向けのアクセス管理サービスであり、 一般消費者向けモバイルアプリのサインアップ機能には適していません。

問3

オンプレミスのアプリケーションサーバーがVPC内のプライベートサブネットにあるインスタンスと定期的に 大容量データをやり取りする必要があり、通信の安定性と帯域を重視したいと考えています。 構築のリードタイムよりも通信品質を優先する場合、最も適切な接続方式はどれですか。

  1. AWS Site-to-Site VPN
  2. AWS Direct Connect
  3. インターネットゲートウェイ経由のパブリックアクセス
  4. NATゲートウェイ経由のアクセス

解答: B

AWS Direct Connectはオンプレミスとの専用線接続で、インターネットを経由しないため通信が安定し、 大容量データ転送に向いています。構築のリードタイムはSite-to-Site VPNより長くなりますが、 設問では通信品質・帯域を優先条件としているためDirect Connectが適切です。NATゲートウェイやインターネットゲートウェイは オンプレミスとの専用接続の手段ではありません。

問4

あるチームが、S3バケットに保存する機密データについて、暗号化キーのローテーションや アクセス許可を独自のキーポリシーで細かく制御したいと考えています。追加コストや管理の手間を許容してでも この要件を満たすには、どの暗号化方式を選ぶべきですか。

  1. SSE-S3（デフォルトのAmazon S3管理キー）
  2. SSE-KMS（AWS KMSキーを使ったサーバーサイド暗号化）
  3. 暗号化を無効化し、アプリケーション側でのみ制御する
  4. ネットワークACLでアクセス制御を行う

解答: B

SSE-KMSはカスタマーマネージドキーを使うことで、キーポリシーによる細かいアクセス制御や ローテーション管理が可能です。SSE-S3はAWSが管理するキーを使うため、利用者側でキーポリシーやローテーション周期を 制御できません。ネットワークACLは暗号化方式ではなくネットワークトラフィックの制御手段です。

問5

ELBにアタッチするTLS証明書をACMで発行し、DNS検証を使って有効化しました。この証明書の有効期限管理について、 正しい記述はどれですか。

  1. 証明書は自動更新されないため、有効期限前に手動で再発行する必要がある
  2. ELBに関連付けられ、DNS検証を使っている場合、ACMが自動的に証明書を更新する
  3. 証明書は一度発行すると失効することはない
  4. 自動更新の対象にするには、証明書をエクスポートしてELBから切り離す必要がある

解答: B

ACMが発行した証明書は、DNS検証を使い、かつELBやCloudFrontなど対応するAWSサービスに関連付けられている場合、 マネージド更新（自動更新）の対象になります。逆に、外部からインポートした証明書やAWS Private CAで直接発行した 一部の証明書は自動更新の対象外です。

問題は本節および1.1〜1.2節で参照した公式情報源（IAM/Organizations/Cognito/VPC/KMS/S3/ACMの各ドキュメント）の内容に基づき作成。
