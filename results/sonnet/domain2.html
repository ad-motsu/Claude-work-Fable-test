はじめに：Resiliencyとこの章の範囲

SAA-C03試験のDomain 2「Design Resilient Architectures」は出題全体の26%を占め、Domain 1に次ぐ比重を持つ領域です。 公式試験ガイドでは、この領域は次の2つのタスクステートメントに分かれています。

  • Task 2.1: スケーラブルで疎結合なアーキテクチャの設計（API、キューイング、マイクロサービス、コンテナ、サーバーレスなど）
  • Task 2.2: 高可用性・耐障害性アーキテクチャの設計（マルチAZ/マルチリージョン、DR戦略、単一障害点の排除など）

この章で扱う「回復性（Resiliency）」とは、障害が発生してもサービスを継続、あるいは短時間で復旧できる能力を指します。 AWSではAvailability Zone（AZ、1つ以上の独立したデータセンター群）を複数使うことで単一データセンター障害に備え、 さらにAWSリージョン（地理的に離れた複数のAZの集合）をまたぐことで、大規模障害や自然災害にも備える、という 段階的な設計アプローチを取ります。この章では、まずマルチAZ/マルチリージョンの基本パターンを扱い、 次に疎結合設計、コンピュート層のスケーリング、ストレージ・データベースの可用性、最後にバックアップ・DR戦略を扱います。

2.1　マルチAZ/マルチリージョンによる高可用性設計

Elastic Load Balancing（ELB）

ELBは複数のAZにまたがるターゲット（EC2インスタンス、コンテナ、IPアドレスなど）にトラフィックを自動的に分散し、 ヘルスチェックで異常なターゲットを検知して正常なターゲットのみにルーティングするマネージドサービスです。 現行世代には次の3種類があります。

  • Application Load Balancer（ALB）：HTTP/HTTPS（レイヤー7）を扱い、パスベース・ホストベースの ルーティングやコンテナ・Lambdaへのルーティングに向く
  • Network Load Balancer（NLB）：TCP/UDP（レイヤー4）の超低レイテンシー・高スループット処理に向く
  • Gateway Load Balancer（GWLB）：サードパーティ製のセキュリティアプライアンス（ファイアウォールなど）への 透過的なトラフィック挿入に使う

ELBはAWS Certificate Managerの証明書と統合してTLS終端を代行でき、EC2 Auto Scalingと連携すると、 起動・終了したインスタンスが自動的に登録・登録解除されます。

Amazon EC2 Auto Scaling

EC2 Auto Scalingは、Auto Scalingグループ単位で最小・最大・希望容量を指定し、需要に応じてインスタンス数を 自動的に増減させるサービスです。複数のAZを指定すると、グループの拡張時にインスタンスをAZ間で均等に配置し、 単一AZの障害からアプリケーションを保護します。EC2のヘルスチェック（および任意のカスタムヘルスチェック）によって 異常なインスタンスを自動検出し、置き換えることで、指定した容量を維持し続けます。

Amazon Route 53のルーティングポリシー

Route 53はDNSレベルでトラフィックを制御するサービスで、目的に応じて複数のルーティングポリシーを選択できます。

ルーティングポリシー用途
シンプル単一リソースへの基本的なルーティング
フェイルオーバーアクティブ/パッシブ構成でのフェイルオーバーを構成する場合
位置情報（Geolocation）ユーザーの地理的位置に基づいてルーティングする場合
地理的近接性（Geoproximity）リソースの位置に基づき、必要に応じてトラフィックの比重を地域間でシフトする場合
レイテンシーベース複数リージョンのリソースの中から、最もレイテンシーが低いリージョンへルーティングする場合
複数値回答最大8個の正常なレコードをランダムに返す場合
加重指定した比率で複数リソースにトラフィックを振り分ける場合（例: カナリアリリース）

マルチリージョン構成での可用性設計では、フェイルオーバールーティングとヘルスチェックを組み合わせて、 プライマリリージョンの異常を検知した際に自動的にセカンダリリージョンへ切り替える構成がよく使われます。

2.2　疎結合アーキテクチャの設計

Amazon SQS・SNS・EventBridgeの使い分け

疎結合設計では、コンポーネント同士を直接呼び出すのではなく、メッセージングサービスを介して非同期に連携させることで、 一方の障害や遅延が他方に波及しないようにします。代表的な3つのサービスは役割が異なります。

  • Amazon SQS：メッセージをキューに保持し、通常は単一のコンシューマーが処理する疎結合パターン向け。 標準キューは「少なくとも1回の配信」を保証し、メッセージ処理中は可視性タイムアウトの間他のコンシューマーから 見えなくなる。処理に失敗し続けるメッセージはデッドレターキューに退避できる
  • Amazon SNS：パブリッシャーが1つのトピックにメッセージを送信すると、購読している複数のエンドポイント （SQS、Lambda、HTTP、Eメール、SMSなど）へ同時に配信する「ファンアウト」パターンに向く
  • Amazon EventBridge：AWSサービス・自社アプリケーション・SaaS製品などから発生するイベントを 受け取り、ルールに基づいてフィルタリング・変換し、複数のターゲットへ振り分けるサーバーレスのイベントバスサービス。 イベント駆動アーキテクチャの中心的な役割を果たす

SQSとSNSを組み合わせる「SNS→複数のSQSキューへのファンアウト」は、1つのイベントを複数の独立した マイクロサービスに並行して処理させたい場合の典型パターンです。

API GatewayとStep Functions

Amazon API Gatewayは、REST APIやWebSocket APIをフルマネージドで公開・管理するサービスで、 Lambdaやその他のAWSサービス、オンプレミスのHTTPエンドポイントをバックエンドとして統合できます。 スロットリングやAPIキー管理、認可（IAM、Cognito、Lambdaオーソライザーなど）も提供します。

AWS Step Functionsは、複数のLambda関数やAWSサービス呼び出しを、ステートマシン（ワークフロー）として 順序立てて実行・調整するオーケストレーションサービスです。リトライやエラーハンドリングをワークフロー定義の中で 宣言的に扱えるため、複雑な非同期処理をマイクロサービス間で疎結合に連携させる際に使われます。

キャッシュ戦略

疎結合・高スケーラビリティな設計では、頻繁に読み取られるデータをキャッシュすることでバックエンドへの負荷を下げます。 代表的な手段として、データベースの前段に配置するインメモリキャッシュ（Amazon ElastiCacheなど）や、 静的・準静的コンテンツをエッジロケーションでキャッシュするAmazon CloudFrontがあります。 キャッシュ戦略の設計では、データの更新頻度と整合性要件に応じて、キャッシュの有効期限（TTL）や キャッシュ無効化の仕組みをあわせて検討する必要があります。

2.3　コンピュート/コンテナ/サーバーレスとスケーリング

EC2 Auto Scalingのスケーリング手法

Auto Scalingグループは、起動テンプレートを設定情報として使い、スケーリングポリシーに基づいてインスタンス数を 調整します。複数のインスタンスタイプやSpot・オンデマンドといった購入オプションを1つのグループ内で組み合わせて コストを最適化することもできます。Spotインスタンスの中断が見込まれる場合は、Capacity Rebalancingにより、 中断リスクが高まったインスタンスを事前に置き換えることも可能です。また、AMIやテンプレートを更新した際の インスタンスリフレッシュ（ローリング更新、カナリア方式のテストを含む）や、起動・終了時にカスタム処理を挟む ライフサイクルフックもサポートされています。

コンテナオーケストレーション：ECSとEKS

Amazon ECSは、コンテナ化されたアプリケーションのデプロイ・管理・スケーリングを行うフルマネージドの コンテナオーケストレーションサービスです。コンテナを実際に動かす「キャパシティ」の選択肢として、 自分でEC2インスタンスを管理するEC2起動タイプと、サーバーの管理・容量計画が不要な サーバーレスのAWS Fargate起動タイプがあります。ECSではタスク定義（アプリケーションの設計図）、 クラスター（実行基盤）、タスク（実行単位）、サービス（長時間稼働するタスクの集合）という概念でアプリケーションを構成します。

Amazon EKSはKubernetesをマネージドで提供するサービスで、Kubernetesの標準APIやエコシステムをそのまま活用したい場合に 選ばれます。ECSとEKSはどちらもEC2起動タイプ・Fargate起動タイプを選べますが、独自のオーケストレーション方式（ECS）か、 業界標準のKubernetes（EKS）かという選択が主な違いです。

AWS Lambdaによるサーバーレスコンピューティング

AWS Lambdaは、サーバーのプロビジョニングや管理を必要としないサーバーレスコンピューティングサービスです。 API Gateway、S3、SQS、EventBridgeなど200を超えるAWSサービスをトリガーとしてコードを呼び出すことができ、 各呼び出しは独立した実行環境で処理されます。トラフィックの増減に応じて実行環境の生成・破棄を自動的に行うため、 利用者側でスケーリングを管理する必要がありません。

CloudFrontによるエッジアクセラレーション

Amazon CloudFrontは、世界中に分散したエッジロケーションを通じて静的・動的コンテンツを配信するCDN（コンテンツ配信網）です。 ユーザーからのリクエストは最もレイテンシーの低いエッジロケーションにルーティングされ、コンテンツがまだキャッシュに なければオリジン（S3バケットやHTTPサーバーなど）から取得されます。オリジンフェイルオーバーを設定すれば、 プライマリオリジンへのリクエストが失敗した場合に自動的にセカンダリオリジンへリクエストを振り替えることもできます。

2.4　ストレージ・データベースの可用性

Amazon S3の耐久性・可用性

S3標準ストレージクラス（S3 Standard、S3 Intelligent-Tiering、S3 Standard-IAなど）は、AWSリージョン内の 最低3つのAZにオブジェクトを冗長的に格納するよう設計されており、1年間で99.999999999%（イレブンナイン）の 耐久性と99.99%の可用性を実現するよう設計されています。これらのクラスは、AZが1つ丸ごと失われても データを維持できるよう設計されています（S3 One Zone-IAのみ単一AZに格納するため、AZ障害時にデータを 失うリスクがある点に注意が必要です）。さらにS3バージョニングを有効にすると、誤削除や上書きからの復旧が容易になります。

Amazon EFS

Amazon EFSは、複数のAZ・複数のコンピュートインスタンス（EC2、ECS、EKS、Lambda、Fargateなど）から 同時にマウントできるサーバーレスのファイルストレージです。リージョナルファイルシステム（推奨）は 複数の地理的に分離されたAZにデータを冗長的に格納し、1つ以上のAZが利用不能になっても継続的な可用性を提供します。 単一AZにのみ格納する「One Zoneファイルシステム」も選択できますが、その場合はそのAZの障害でデータを 失う可能性があります。

Amazon RDSのマルチAZ配置とリードレプリカ

RDSのマルチAZ配置には、スタンバイが1つの「マルチAZ DBインスタンス配置」と、スタンバイが2つの 「マルチAZ DBクラスター配置」があります。前者のスタンバイは同期レプリケーションによるフェイルオーバー用途のみで 読み取りトラフィックは処理しませんが、後者はフェイルオーバー用途に加えて読み取りトラフィックも処理できます。

これに対してリードレプリカは、読み取り性能をスケールするための仕組みで、 非同期レプリケーションによって作成される読み取り専用のコピーです。マルチAZ配置が「同一構成の可用性確保」を 目的とするのに対し、リードレプリカは「読み取り負荷の分散」を主目的とし、リージョンをまたいで作成することもできます。 Amazon Auroraでは、Aurora Global Databaseを使うことで、セカンダリリージョンへ通常1秒未満のレイテンシーで レプリケーションでき、障害時にはセカンダリリージョンを1分未満で書き込み可能な状態に昇格できます。

Amazon RDS Proxy

RDS Proxyは、アプリケーションとデータベースの間でコネクションプーリングを行うフルマネージドのプロキシです。 Lambdaのように多数の短命な接続を大量に生成しがちなサーバーレスアプリケーションから利用すると、 データベースへの接続数の急増によるオーバーサブスクリプションを防げます。また、プライマリDBインスタンスに障害が 発生した際は、アプリケーション側の接続を維持したまま自動的にスタンバイDBインスタンスへの接続に切り替えるため、 フェイルオーバーの影響を受けにくくなります。

Amazon DynamoDBの可用性

DynamoDBはマネージドのNoSQLデータベースで、データを自動的に複数のAZにレプリケートすることで高い可用性と 耐久性を実現しています。単一リージョン内での可用性はサービスに組み込まれており、利用者が明示的にマルチAZ配置を 構成する必要はありません。複数のAWSリージョンにまたがる可用性やDR、低レイテンシーなグローバルアクセスが 必要な場合は、Global Tables機能を使うことで、複数リージョン間で双方向レプリケーションされたテーブルを構成できます。

2.5　バックアップ・DR戦略

AWS Backup

AWS Backupは、EC2/EBS、RDS（Auroraを含む）、DynamoDB、EFS、Storage Gateway、FSxなど複数のAWSサービスにまたがる バックアップを一元的に設定・スケジュール・監視できるフルマネージドサービスです。「バックアッププラン」で バックアップ頻度や保持期間、コールドストレージへのライフサイクル移行ルールを定義し、タグベースでリソースに 適用できます。クロスリージョン・クロスアカウントでのバックアップコピーもサポートしており、 リージョン障害やアカウント侵害に備えたバックアップの分散保管が可能です。バックアップボールトには 「Vault Lock」機能があり、WORM（Write Once Read Many）モデルでバックアップの削除・変更を防止できます。

DR戦略の4類型とRTO/RPO

AWSの公式ホワイトペーパーでは、災害復旧（DR）戦略はコストと複雑さが低い順に、次の4つに大別されています。 RTO（目標復旧時間）はどれだけ早く復旧できるか、RPO（目標復旧時点）はどれだけのデータ損失まで許容できるかを表す指標です。

戦略概要RTOの傾向コストの傾向
バックアップ&リストア データを定期的にバックアップし、障害時にインフラをゼロから再構築して復元する 数時間〜 最も低い
パイロットライト データベースなどコア基盤は常時稼働・レプリケーションしておき、アプリケーションサーバーなどは 平常時は停止しておいて障害時に起動・スケールする 数十分〜 低〜中
ウォームスタンバイ 縮小版ではあるが常時稼働している本番相当の環境をセカンダリリージョンに用意し、 障害時にスケールアウトして本番容量まで引き上げる 数分〜 中〜高
マルチサイトActive/Active 複数リージョンで同時に本番トラフィックを処理し、いずれかのリージョン障害時も他のリージョンで 処理を継続する ほぼゼロ（データ破損時はゼロではない） 最も高い

戦略を選ぶ際は、単一データセンター（AZ）レベルの障害であればバックアップ&リストアで十分な場合が多い一方、 リージョン単位の障害や規制要件がある場合はパイロットライト以上を検討する、という判断基準が示されています。 フェイルオーバーの実行には、Route 53のヘルスチェックとフェイルオーバールーティング、またはAWS Global Acceleratorの エンドポイント切り替えなどが使われます。

AWS X-Rayによる可観測性

AWS X-Rayは、分散アプリケーションが処理するリクエストに関するデータを収集し、可視化・分析するサービスです。 リクエストがマイクロサービスやデータベース、外部APIなど複数のコンポーネントをまたいで処理される様子を 「トレースマップ」として可視化でき、ボトルネックやレイテンシーの急増箇所を特定するのに役立ちます。 可用性設計においては、単一障害点や遅延の原因となっているコンポーネントを特定するための観測手段として位置づけられます。

各設問は4択の単一選択式です。解答を考えてから、折りたたみ内の解説を確認してください。

問1

あるWebアプリケーションで、複数リージョンにデプロイした環境のうち、最もレイテンシーが低いリージョンへ ユーザーを振り分けたいと考えています。最も適したRoute 53のルーティングポリシーはどれですか。

  1. シンプルルーティング
  2. 加重ルーティング
  3. レイテンシーベースルーティング
  4. 複数値回答ルーティング

解答: C

レイテンシーベースルーティングは、複数リージョンに配置されたリソースの中から、ユーザーにとって最もレイテンシーが 低いリージョンへトラフィックを振り分けるためのポリシーです。加重ルーティングは指定した比率での振り分け、 複数値回答はランダムな複数レコードの返却に使われ、いずれもレイテンシー最適化を目的としたものではありません。

問2

注文処理システムで、1件の注文イベントを「在庫更新」「請求処理」「配送手配」という3つの独立したマイクロサービスに 同時に配信し、それぞれが非同期に処理を行えるようにしたいと考えています。最も適したAWSサービスの組み合わせはどれですか。

  1. Amazon SQS標準キューを1つだけ使い、3つのサービスが同じキューをポーリングする
  2. Amazon SNSトピックに注文イベントを発行し、3つのSQSキューをそれぞれサブスクライブさせるファンアウト構成にする
  3. 3つのマイクロサービスを直接同期的なHTTP呼び出しでチェーンする
  4. Amazon RDSのテーブルにポーリング用のフラグ列を追加し、各サービスが定期的に参照する

解答: B

SNSトピックに発行したメッセージを複数のSQSキューにファンアウトする構成は、1つのイベントを複数の独立した コンシューマーに同時配信する典型的な疎結合パターンです。単一のSQSキューを複数サービスで共有すると、 1件のメッセージは通常いずれか1つのコンシューマーにしか処理されないため要件を満たしません。

問3

Lambda関数からRDS for MySQLへの接続数が急増し、データベース側の最大接続数に達してエラーが発生するようになりました。 アプリケーションコードの大幅な変更を避けつつ、この問題を軽減する最も適切な方法はどれですか。

  1. RDSインスタンスをマルチAZ配置に変更する
  2. RDS Proxyを導入し、Lambdaからの接続をコネクションプール経由にする
  3. DynamoDBに移行する
  4. Route 53のレイテンシーベースルーティングを設定する

解答: B

RDS Proxyはコネクションプーリングを行い、多数の短命な接続を生成しがちなLambdaのようなサーバーレス環境からの 接続を効率的に再利用します。これによりデータベース側の接続数オーバーサブスクリプションを緩和できます。 マルチAZ配置は可用性の向上が目的であり、接続数の問題そのものは解決しません。

問4

規制要件により、災害の定義がリージョン単位の障害にまで及ぶ企業が、コストを抑えつつコアデータベースは 常時レプリケーションしておき、アプリケーションサーバーは平常時は起動せず障害時のみ起動・スケールする DR戦略を採用したいと考えています。最も適切な戦略はどれですか。

  1. バックアップ&リストア
  2. パイロットライト
  3. ウォームスタンバイ
  4. マルチサイトActive/Active

解答: B

パイロットライトは、データベースなどのコア基盤を常時稼働・レプリケーションしておきつつ、 アプリケーションサーバーなど他の要素は平常時「switched off」にしておき、障害時に起動・スケールする戦略です。 バックアップ&リストアより復旧が速く、ウォームスタンバイほどのコストはかからない中間的な選択肢です。

問5

S3標準ストレージクラスに保存したオブジェクトの耐久性設計に関する記述として、正しいものはどれですか。

  1. 単一のAZにのみ保存され、AZ障害時にはデータが失われる可能性がある
  2. 最低3つのAZにまたがって冗長的に保存されるよう設計されており、AZが1つ丸ごと失われてもデータを維持できるよう設計されている
  3. 耐久性は保証されておらず、利用者が独自にレプリケーションを構成する必要がある
  4. バージョニングを有効にしないと複数AZへの保存は行われない

解答: B

S3標準（S3 Standard）などの主要ストレージクラスは、リージョン内の最低3つのAZに複数デバイスにわたって 冗長的にオブジェクトを保存するよう設計されており、99.999999999%の耐久性を実現するよう設計されています。 これはバージョニングの有効・無効に関わらないデフォルトの冗長化の仕組みです。単一AZにのみ保存されるのは S3 One Zone-IAストレージクラスの特性です。

問題は本節および2.1〜2.5節で参照した公式情報源（Route53/SQS・SNS/RDS Proxy/DR whitepaper/S3の各ドキュメント）の内容に基づき作成。
