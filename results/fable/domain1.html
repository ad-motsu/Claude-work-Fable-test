1.1　IAMとアクセス管理

Task Statement 1.1: Design secure access to AWS resources.

IAM の4つの構成要素

IAM（Identity and Access Management）は AWS 内の「誰が」「何を」できるかを決める仕組みです。実務で名前は知っていても、試験では各要素の使い分けの判断が問われます。

要素役割設計上のポイント
ユーザー永続的な資格情報を持つ個人・人間の ID長期のアクセスキーは極力避ける。人間には後述の Identity Center を推奨
グループユーザーをまとめて権限を付与する箱権限はグループに付け、個人ユーザーへの直接付与を減らすと管理が楽
ロール一時的な認証情報を「引き受ける（assume）」仕組み。EC2・Lambda・他アカウント・フェデレーションユーザーが使うアクセスキーの埋め込みを不要にする鍵。試験の頻出テーマ
ポリシー許可/拒否を JSON で定義したルールアイデンティティベースとリソースベースの2系統がある（後述）
実務の「知っている」から一歩進める

EC2 インスタンスから S3 を読ませたいとき、アクセスキーをコードや環境変数に置くのは設計ミスと判断されます。正解は IAM ロール をインスタンスプロファイル経由でアタッチし、EC2 が STS から一時認証情報を自動取得する構成です。「資格情報がハードコードされている／ローテーションされない」という選択肢は、ほぼ不正解の距離にあります。

ポリシー評価ロジック — 試験の最重要トピック

複数のポリシーが同時に効いているとき、最終的な許可/拒否がどう決まるかは頻出です。基本ルールは次の順序です。

  1. 明示的な Deny が1つでもあれば、即座に拒否（最優先。他の Allow を必ず上書きする）
  2. 明示的な Deny がなく、いずれかのポリシーに Allow があれば許可
  3. どこにも Allow がなければ 暗黙的な拒否（デフォルト Deny）

合言葉は「明示的 Deny ＞ 明示的 Allow ＞ 暗黙的 Deny」です。加えて、評価に関わるポリシーは1種類ではありません。

ポリシーの種類何に付くか典型的な用途
アイデンティティベースポリシーユーザー/グループ/ロール「この ID は何ができるか」を定義
リソースベースポリシーS3バケット、SQS、KMSキー等のリソース側クロスアカウントアクセスや、誰にリソースを開くかを定義
SCP（サービスコントロールポリシー）Organizations の OU/アカウントアカウント全体の権限の上限（ガードレール）を設定。許可は付与せず、上限を絞るだけ
権限境界（Permissions Boundary）ユーザー/ロールそのIDが持てる権限の最大範囲を制限（委任時の暴走防止）
セッションポリシーAssumeRole 時に一時付与一時セッションの権限をさらに絞る
よくある勘違い

SCP は「権限を付与する」ものではありません。SCP はアカウント内で行使できる権限の天井を決めるだけで、実際の許可は依然として IAM ポリシーが必要です。SCP で Deny された操作は、そのアカウントでは管理者（root を含む）でも実行できなくなります。

ロールと STS、クロスアカウントアクセス

ロールの本質は「AWS STS（Security Token Service）が発行する一時的な認証情報を引き受ける」ことです。長期キーと違い、有効期限つきで自動失効するため、漏えい時のリスクが小さくなります。試験で出るパターンは主に次の3つです。

  • サービスロール: EC2 / Lambda / ECS タスクなどが AWS API を呼ぶために引き受ける
  • クロスアカウントアクセス: 本番アカウントのロールを、開発アカウントのユーザーが sts:AssumeRole で引き受ける。信頼ポリシーで「誰が引き受けてよいか」を定義する
  • ロールスイッチ: 1人のユーザーが複数アカウント間を切り替えて作業する

フェデレーションと IAM Identity Center

社内の既存 ID（Active Directory、Okta、Google Workspace など）で AWS にサインインさせたい場合、AWS に個別ユーザーを作らずフェデレーションします。仕組みは「外部 IdP で認証 → SAML/OIDC のトークン → IAM ロールを引き受ける」という流れです。

複数アカウントに人間がアクセスする現代的な推奨は IAM Identity Center（旧 AWS SSO）です。1か所で ID を管理し、アカウント×権限セットの割り当てを一元化できます。「多数のアカウントに大勢の従業員がアクセスする。IAM ユーザーを各アカウントに作りたくない」という要件が出たら、Identity Center が有力な解になります。

マルチアカウント戦略：Organizations / SCP / Control Tower

組織が成長するとアカウントは増えます。それらを束ねて統制するのが AWS Organizations です。

  • Organizations: 複数アカウントを OU（組織単位）で階層管理し、一括請求（Consolidated Billing）も担う
  • SCP: OU/アカウント単位でガードレールを適用（例：「特定リージョン以外の利用を全社禁止」）
  • AWS Control Tower: ベストプラクティスに沿ったマルチアカウント環境（ランディングゾーン）を自動構築・統制。裏で Organizations と SCP を使う

アクセスに関するベストプラクティス

  • 最小権限の原則: 必要な操作だけを許可し、広い * 権限は避ける
  • root ユーザーの保護: 日常運用に使わず、MFA を有効化し、アクセスキーは作らない
  • MFA の徹底: 特権操作・root には多要素認証を必須にする
  • 長期アクセスキーを避ける: 人間は Identity Center、ワークロードはロールで一時認証情報を使う

1.2　ワークロードとネットワークのセキュリティ

Task Statement 1.2: Design secure workloads and applications.

VPC のセキュリティ構成要素：SG と NACL の使い分け

VPC 内のトラフィック制御は、主にセキュリティグループ（SG）とネットワーク ACL（NACL）の2層で行います。両者の違いは最頻出です。

観点セキュリティグループ（SG）ネットワーク ACL（NACL）
適用先ENI（インスタンス）単位サブネット単位
状態ステートフル（戻りの通信は自動許可）ステートレス（戻りの通信も明示的に許可が必要）
ルール許可（Allow）のみ。拒否は書けない許可と拒否の両方を書ける
評価すべてのルールを総合評価番号順に評価し、最初に一致したルールを適用
用途インスタンスへの通信を細かく制御（第一の砦）サブネット全体で特定 IP の遮断など補完的に使う
判断のコツ

「特定の IP アドレスからのアクセスを明示的に拒否（ブロック）したい」という要件なら、Deny を書ける NACL が答えです。SG は許可ルールしか持てないため、拒否そのものを表現できません。

ネットワークセグメンテーション

基本設計は「公開が必要なものだけをパブリックサブネットに、それ以外はプライベートサブネットに」です。ロードバランサーや踏み台はパブリック側、アプリケーションサーバーや DB はプライベート側に置き、プライベートサブネットからのアウトバウンド通信は NAT ゲートウェイ経由にします。これによりインターネットから DB へ直接到達する経路をなくせます。

脅威対策：WAF・Shield・DDoS

サービス守る層要点
AWS WAFアプリケーション層（L7）SQL インジェクションや XSS、不正な HTTP リクエストをルールでフィルタ。CloudFront / ALB / API Gateway に適用
AWS Shield Standardネットワーク／トランスポート層（L3/L4）全 AWS 利用者に自動・無料で適用される基本的な DDoS 防御
AWS Shield AdvancedL3/L4/L7有料サブスクリプション。高度な DDoS 検知、専門チーム（SRT）の支援、攻撃起因のスケール費用の補償を提供
組み合わせで考える

「L7 の SQL インジェクションを防ぎたい」なら WAF、「大規模 DDoS への追加保護と費用補償が欲しい」なら Shield Advanced。両者は排他ではなく、CloudFront + WAF + Shield Advanced のように多層で重ねるのが定石です。

脅威の検知・可視化サービス

「予防」だけでなく「検知」も設計に含めます。名前が似ていて紛らわしいので、何を入力に、何を見つけるかで覚えます。

サービス役割覚え方
Amazon GuardDuty脅威検知。VPCフローログ・DNSログ・CloudTrail等を機械学習で分析し、不審な挙動を検出「悪意ある通信・侵害の兆候」を見張る番犬
Amazon MacieS3 内の機密データ（個人情報・認証情報など）を検出・分類データの中身を見る（Macie ＝ 機密データ）
Amazon InspectorEC2・コンテナイメージ・Lambda の脆弱性を自動スキャンパッチ未適用や CVE を点検
AWS Security Hub各種セキュリティ結果を集約し、ベストプラクティス基準で評価（CSPM）セキュリティ状態のダッシュボード

アプリの資格情報管理：Secrets Manager と Parameter Store

DB パスワードや API キーをコードに書かないための選択肢が2つあります。

  • AWS Secrets Manager: シークレット専用。自動ローテーション（RDS 等と連携してパスワードを定期更新）を組み込みで持つ。有料
  • SSM Parameter Store: 設定値やシークレットを保存。標準パラメータは無償で、SecureString で KMS 暗号化も可能。ただし自動ローテーションは標準機能ではない
判断のコツ

「DB 認証情報を定期的に自動ローテーションしたい」なら Secrets Manager。「単に設定値やキーを安全に保存したい／コストを抑えたい」なら Parameter Store。ローテーション要件の有無が分岐点です。

安全なアプリアクセスとエンドポイント

  • Amazon Cognito: Web/モバイルアプリのユーザー認証（サインアップ/サインイン、SNS・SAML フェデレーション）を担うマネージド ID サービス
  • VPC エンドポイント / AWS PrivateLink: S3 や DynamoDB、他サービスへの通信をインターネットを経由せず AWS 内部ネットワークで完結させる。ゲートウェイ型（S3/DynamoDB）とインターフェース型（PrivateLink）がある

ハイブリッド接続のセキュリティ

接続方式特徴
Site-to-Site VPNインターネット経由の暗号化トンネル。低コスト・短時間で開始できるが、帯域と遅延はインターネット依存
AWS Direct ConnectオンプレとAWSを結ぶ専用線。安定した帯域・低遅延。ただし単体では暗号化されないため、機密要件では上に IPsec VPN を重ねる構成もとる

1.3　データ保護と暗号化

Task Statement 1.3: Determine appropriate data security controls.

AWS KMS とエンベロープ暗号化

保管中データの暗号化は AWS KMS（Key Management Service）が中心です。KMS の鍵（KMS キー）は3種類あります。

鍵の種類管理主体・特徴
AWSマネージドキーサービス（aws/s3 など）が自動生成。利用者は削除・ポリシー変更ができない。年1回自動ローテーション
カスタマー管理キー（CMK）利用者が作成・管理。キーポリシーやローテーション、削除を制御できる。監査やコンプライアンス要件で使う
AWS所有キーAWS が複数顧客向けに保持。利用者からは見えない

KMS はエンベロープ暗号化を使います。実データは高速なデータキーで暗号化し、そのデータキー自体を KMS キーで暗号化して一緒に保管します。KMS キーは KMS の外に出ないため、鍵の安全性を保ちながら大量データを効率よく暗号化できます。

キーのローテーション（2026年時点の仕様）

  • カスタマー管理キーは自動ローテーションを有効化でき、ローテーション間隔は 90〜2560日の範囲で設定可能（デフォルトは約365日）。オンデマンドのローテーションも手動で実行できる
  • ローテーションしてもキー ID は変わらず、古い鍵素材は過去データの復号のため保持される。アプリ側の参照を変更する必要はない
  • AWSマネージドキーは年1回自動でローテーションされる

保管中の暗号化：S3 と EBS（デフォルト挙動に注意）

2026年時点の最新仕様 — S3 はデフォルトで暗号化される

Amazon S3 は2023年1月以降、すべての新規オブジェクトを既定で暗号化します。何も指定しなければ SSE-S3（AES-256、AWS 管理の鍵、追加料金なし）が自動適用されます。したがって「S3 の保管データが暗号化されていない」という前提の選択肢は、現在の仕様では基本的に成り立ちません。

S3 の暗号化方式（サーバーサイド暗号化）は用途で選びます。

方式鍵の管理使いどころ
SSE-S3AWS が管理（AES-256）既定。特別な鍵要件がなければこれで十分
SSE-KMSKMS キー（監査ログ・アクセス制御・ローテーション）誰がいつ復号したかを CloudTrail で追跡したい、鍵アクセスを細かく制御したい場合
DSSE-KMSKMS による二重暗号化規制で多層暗号化が要求される場合
SSE-C利用者が鍵を持ち込み、AWS は保持しない鍵を自社管理し AWS に預けたくない場合

Amazon EBS はボリューム暗号化に KMS を使います。既定では新規ボリュームは暗号化されませんが、リージョン単位のアカウント設定で「デフォルト暗号化」を有効にすると、以降の新規ボリュームとスナップショットが自動的に暗号化されます。暗号化ボリュームから作ったスナップショット、その復元ボリュームも暗号化が引き継がれます。

転送中の暗号化と ACM

転送中データは TLS（HTTPS）で保護します。証明書の管理は AWS Certificate Manager（ACM）が担います。

  • ACM が発行したパブリック証明書は自動更新（renewal）される。期限切れによる障害を避けられる
  • ACM 証明書は ELB・CloudFront・API Gateway などに直接関連付けて TLS を終端できる（EC2 に直接インストールする形では使えない点に注意）
  • KMS の「鍵のローテーション」と ACM の「証明書の更新」は、いずれも失効・漏えいリスクを下げる定期更新という同じ発想

データの保護・保持・復旧

  • バックアップとレプリケーション: S3 のクロスリージョンレプリケーション、RDS スナップショット、AWS Backup による一元的なバックアップ（第2章で詳述）
  • ライフサイクルとデータ分類: S3 ライフサイクルで古いデータを低コストストレージへ移行・失効。機密度の分類には Macie を併用
  • アクセスとガバナンス: S3 のブロックパブリックアクセス、バケットポリシー、KMS キーポリシーで「誰が復号できるか」まで含めて制御する

章末問題

4択・全5問。各問の「解答と解説」を開く前に、まず自分で選んでみてください。

問1. あるユーザーに対し、アイデンティティベースポリシーは特定の S3 バケットへの s3:GetObject を明示的に許可している。一方、同じバケットのバケットポリシーは、そのユーザーのプリンシパルに対して s3:GetObject を明示的に拒否している。このユーザーはオブジェクトを取得できるか。

  • A. 取得できる。アイデンティティベースポリシーの許可が優先されるため
  • B. 取得できない。明示的な拒否はすべての許可を上書きするため
  • C. 取得できる。バケットポリシーよりユーザーポリシーが強いため
  • D. どちらのポリシーが後に作成されたかで決まる
解答と解説を見る

正解 B

IAM の評価では「明示的 Deny ＞ 明示的 Allow ＞ 暗黙的 Deny」が絶対のルールです。どのポリシー（アイデンティティベース／リソースベース）に書かれていても、明示的な Deny が1つでもあれば結果は拒否になります。ポリシーの作成順序（D）や種類による優劣（A・C）は関係ありません。

問2. 数百のAWSアカウントを持つ企業が、全アカウントで「承認された特定リージョン以外のリソース作成を組織的に禁止」したい。管理者権限を持つ利用者であっても回避できないようにする。最も適切な仕組みはどれか。

  • A. 各アカウントの全 IAM ユーザーに拒否ポリシーを個別付与する
  • B. AWS Organizations のサービスコントロールポリシー（SCP）で対象リージョンを拒否する
  • C. 各アカウントで NACL を設定してリージョンを制限する
  • D. IAM Identity Center の権限セットでリージョンを制限する
解答と解説を見る

正解 B

SCP はアカウント全体の権限の上限（ガードレール）で、そのアカウントの管理者や root であっても SCP の Deny を超えて操作できません。全アカウントに一括適用でき、スケールします。A は数百アカウントでは非現実的かつ抜け漏れが生じます。NACL（C）はネットワークトラフィック制御であり API のリージョン制限には使えません。Identity Center の権限セット（D）は Identity Center 経由のアクセスにしか効かず、アカウント全体の強制力はありません。

問3. EC2 上のアプリケーションから S3 バケットのオブジェクトを読み取る必要がある。セキュリティのベストプラクティスに最も沿った認証情報の扱いはどれか。

  • A. IAM ユーザーのアクセスキーを作成し、アプリの環境変数に設定する
  • B. アクセスキーをソースコードに埋め込み、暗号化してデプロイする
  • C. S3 読み取り権限を持つ IAM ロールを EC2 インスタンスにアタッチする
  • D. root ユーザーのアクセスキーをインスタンスに配置する
解答と解説を見る

正解 C

EC2 に IAM ロールをアタッチすると、アプリは STS が発行する有効期限つきの一時認証情報を自動取得でき、長期キーをどこにも保存せずに済みます。ローテーションも AWS が自動で行います。A・B は長期キーの保存・漏えいリスクがあり、ローテーションもされません。D は root キーの利用そのものが重大なアンチパターンです。

問4. Web アプリケーションが SQL インジェクションや XSS などアプリケーション層（L7）の攻撃を受けている。CloudFront/ALB の前段でこれらの不正なリクエストをルールベースでフィルタするのに最も適したサービスはどれか。

  • A. AWS Shield Standard
  • B. AWS WAF
  • C. セキュリティグループ
  • D. Amazon GuardDuty
解答と解説を見る

正解 B

AWS WAF は L7 の Web リクエストを検査し、SQL インジェクションや XSS、悪意あるパターンをルールで遮断します。Shield Standard（A）は L3/L4 の DDoS を自動防御しますが、L7 のインジェクション対策ではありません。セキュリティグループ（C）は IP/ポート単位の許可制御で、リクエスト内容は見られません。GuardDuty（D）は脅威を検知するサービスで、リクエストをインラインでブロックする役割ではありません。

問5. RDS データベースの認証情報を安全に保管し、かつ定期的に自動でローテーションしたい。最も適したサービスはどれか。

  • A. SSM Parameter Store（標準パラメータ）に平文で保存する
  • B. AWS Secrets Manager に保存し、自動ローテーションを構成する
  • C. S3 バケットに暗号化して保存し、手動で更新する
  • D. IAM ポリシーに認証情報を記述する
解答と解説を見る

正解 B

AWS Secrets Manager はシークレット専用サービスで、RDS などと連携した認証情報の自動ローテーションを組み込みで提供します。Parameter Store（A）はシークレット保存はできますが、標準では自動ローテーション機能を持たず、平文保存は論外です。S3 への保存＋手動更新（C）は自動化要件を満たしません。IAM ポリシーは権限定義であり、認証情報の保管場所ではありません（D）。
