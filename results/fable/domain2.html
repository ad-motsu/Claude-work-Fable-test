はじめに

Domain 2 は採点対象の 26% を占め、Domain 1 に次ぐ配点です。

このドメインのキーワードは Resilient（弾力性・回復力）です。単に「壊れないシステム」ではなく、コンポーネントが壊れることを前提に、それでもサービスを継続できる／自動で立ち直れる設計を問います。混同しやすい2つの概念を最初に整理します。

概念意味実現手段の例
高可用性（HA）障害が起きても短いダウンタイムでサービスを継続できるマルチAZ、ロードバランサ、Auto Scaling、フェイルオーバー
耐障害性（Fault Tolerance）コンポーネント障害時もダウンタイムなく動き続ける冗長化＋自動復旧、複数AZ/リージョンへの多重化
スケーラビリティ負荷の増減に応じて容量を伸縮できる水平スケール（Auto Scaling）、疎結合（SQS等）
疎結合（Loose Coupling）コンポーネント間の依存を弱め、一部の障害を全体に波及させないキュー（SQS）、通知（SNS）、イベント（EventBridge）
前提知識：AWS グローバルインフラの階層

リージョンは地理的に離れた独立エリア（例：東京、大阪）。各リージョンは複数のアベイラビリティゾーン（AZ）で構成され、AZ同士は物理的に分離されつつ低遅延の専用線で結ばれています。可用性設計の基本は「複数AZにまたがって冗長化する（Multi-AZ）」こと。1つのAZが丸ごと落ちても別のAZで受け続けられます。さらに広域災害に備えるならリージョンをまたぎます。

2.1　スケーラブルで疎結合なアーキテクチャ

Task Statement 2.1: Design scalable and loosely coupled architectures.

スケーリングの2方向

  • 垂直スケール（スケールアップ）: 1台のスペックを上げる。単純だが上限があり、変更時に停止・再起動を伴いやすく、単一障害点も残る
  • 水平スケール（スケールアウト）: 台数を増やす。クラウドの基本。負荷分散と組み合わせ、弾力性・可用性の両方に効く

試験では原則として水平スケール＋疎結合が「クラウドらしい正解」の方向です。

Elastic Load Balancing（ELB）：4種類の使い分け

ロードバランサは複数ターゲットにトラフィックを分散し、異常なターゲットをヘルスチェックで切り離します。SAAでは種類の選択が頻出です。

種類レイヤー主な用途・特徴
ALB（Application）L7（HTTP/HTTPS）パスやホスト名によるルーティング、コンテナ/マイクロサービス、WAF連携。Webアプリの定番
NLB（Network）L4（TCP/UDP/TLS）超低遅延・超高スループット、静的IP/Elastic IP、極端な同時接続数。TCPレベルの負荷分散
GWLB（Gateway）L3/L4ファイアウォール等のサードパーティ仮想アプライアンスを透過的に挿入
CLB（Classic）L4/L7旧世代。新規設計では非推奨（ALB/NLBを選ぶ）
判断のコツ

「HTTPのパスで /api と /img を別ターゲットに振り分けたい」→ ALB。「TCPで固定IPと極低遅延が要る」「毎秒数百万接続」→ NLB。まず L7 か L4 かで切り分けます。

Amazon EC2 Auto Scaling

Auto Scaling グループ（ASG）は、需要に応じてEC2台数を自動で増減させます。可用性の観点でも重要で、複数AZにまたがってインスタンスを分散配置し、異常なインスタンスを終了して健全なものを自動起動します。

  • 希望/最小/最大キャパシティで台数の範囲を定義
  • スケーリングポリシー：ターゲット追跡（例：平均CPU 50%を維持）、ステップ、スケジュール、予測スケーリング
  • ELBのヘルスチェックと連携し、異常インスタンスを置き換える（自己修復）

ALB ＋ Auto Scaling ＋ マルチAZは、スケーラブルかつ高可用なWeb層の王道パターンです。

疎結合：SQS / SNS / EventBridge

コンポーネントを直接呼び合わせると、片方の障害や過負荷がもう片方に波及します。間にメッセージング層を挟んで疎結合にするのがレジリエンス設計の核心です。

サービスモデル使いどころ
Amazon SQSキュー（1対1、ポーリング型）処理を非同期化しバッファリング。生産者と消費者を切り離し、急なスパイクを吸収。消費側をASGでスケール
Amazon SNSPub/Sub（1対多、プッシュ型）1つのイベントを複数の購読者へ同時配信（ファンアウト）
Amazon EventBridgeイベントバス（ルーティング）多数のAWSサービス/SaaSのイベントをルールで振り分け。イベント駆動アーキテクチャの中核
SQSの2タイプと信頼性の仕掛け

標準キューはほぼ無制限のスループットで「少なくとも1回配信（順序はベストエフォート、稀に重複）」、FIFOキューは「厳密な順序・重複排除」。処理に失敗し続けるメッセージはデッドレターキュー（DLQ）に隔離して、詰まりや無限リトライを防ぎます。SQS＋Auto Scaling（キューの深さに応じて消費者を増減）は疎結合スケーリングの定番です。

サーバーレスとコンテナによる弾力性

  • AWS Lambda: サーバー管理なしでコードを実行。リクエストに応じて自動でスケールし、アイドル時のコストがない。イベント駆動の中心
  • AWS Fargate: ECS/EKSのサーバーレス実行基盤。EC2の管理なしでコンテナを動かす
  • ECS / EKS: コンテナのオーケストレーション。ECSはAWSネイティブでシンプル、EKSはKubernetes互換
  • API Gateway: フルマネージドなAPIフロント。スロットリング（流量制御）でバックエンドを保護
  • AWS Step Functions: 複数の処理をステートマシンとして編成し、リトライ・エラーハンドリングを宣言的に組む

2.2　高可用・耐障害アーキテクチャ

Task Statement 2.2: Design highly available and/or fault-tolerant architectures.

Route 53：DNSによる可用性設計

Amazon Route 53 は権威DNSであり、ルーティングポリシーとヘルスチェックを組み合わせて可用性に貢献します。

ルーティングポリシー用途
フェイルオーバープライマリが不健全になったらセカンダリ（DRサイト等）へ自動切替。アクティブ/パッシブDRの要
レイテンシーベース最も低遅延のリージョンへ誘導
加重（Weighted）重み付けで割合分散。カナリアリリース等
位置情報 / 地理的近接ユーザーの地理的位置に基づく振り分け
複数値回答健全な複数エンドポイントを返し簡易的に分散

コンピュート層の高可用パターン

  • 単一障害点（SPOF）の排除: 1台構成・1AZ構成をやめ、複数AZに冗長化する
  • ALB + ASG + マルチAZ: Web/アプリ層の標準。AZ障害時も残るAZで継続
  • NATゲートウェイの冗長化: 各AZにNATゲートウェイを置き、片方のAZ障害でアウトバウンドが全断しないようにする
  • ステートレス設計: セッション情報はインスタンス外（ElastiCache/DynamoDB）に持たせ、どのインスタンスが落ちても影響しないようにする

データベースの可用性：Multi-AZ とリードレプリカ

最頻出の区別：Multi-AZ ≠ リードレプリカ

この2つは目的が違います。混同すると設問を落とします。

機能目的挙動
RDS Multi-AZ可用性（HA/DR）別AZに同期レプリケーションのスタンバイを保持。障害時に自動フェイルオーバー。スタンバイは読み取りには使えない（待機専用）
リードレプリカ読み取りのスケール（性能）非同期レプリケーション。読み取りクエリを分散。別リージョンにも作成可。それ自体は自動フェイルオーバーの仕組みではない

「読み取り負荷が高くて遅い」→ リードレプリカ。「AZ障害でもDBを継続させたい」→ Multi-AZ。「両方」なら併用します。

Amazon Aurora

  • ストレージが3つのAZに各2つ、計6コピー複製され、高い耐久性を持つ
  • 読み取り用のAuroraレプリカを最大15台まで追加でき、プライマリ障害時は自動昇格でフェイルオーバー
  • Aurora Global Databaseで複数リージョンに低遅延複製し、リージョン規模の障害に備える

Amazon DynamoDB

  • フルマネージドなNoSQL。データは単一リージョン内で自動的に複数AZへ複製され、標準で高可用
  • グローバルテーブルで複数リージョンにアクティブ/アクティブ複製し、マルチリージョンの可用性を実現
  • ポイントインタイムリカバリ（PITR）で直近35日間の任意時点に復元可能

ストレージの可用性・耐久性

サービス可用性・耐久性の特徴
Amazon S3オブジェクトを複数AZに冗長格納し、標準クラスで 99.999999999%（イレブンナイン）の耐久性を設計目標とする。バージョニングで誤削除・上書きから保護。クロスリージョンレプリケーション（CRR）で地理冗長
Amazon EFS複数AZに分散する共有ファイルストレージ。複数のEC2/AZから同時マウントでき、それ自体がリージョン内で冗長
Amazon EBSブロックストレージ。単一AZ内で自動複製される（AZをまたがない）。可用性はスナップショット（S3保管）＋別AZでの復元で担保
Amazon FSxWindows/Lustre等のマネージドファイル。Multi-AZ構成が可能
判断のコツ

「複数AZのEC2から共有ファイルシステムを同時に読み書きしたい」→ EBS（単一AZ・単一アタッチが基本）ではなく EFS。EBSは1つのAZに紐づく点が可用性設計の落とし穴になりやすいです。

2.3　バックアップとディザスタリカバリ（DR）

Task Statement 2.2 の一部：DR戦略・RTO/RPO・データ耐久性。

RTO と RPO — DR設計の2つの物差し

指標意味覚え方
RTO（Recovery Time Objective）障害発生から復旧までの目標時間。「どれだけ早く戻すか」Time＝ダウンタイムの許容量
RPO（Recovery Point Objective）復旧時に許容できるデータ損失量を時間で表す。「どこまでのデータ喪失を許すか」Point＝直近バックアップからの巻き戻り幅

RTO/RPOを小さくするほど可用性は上がりますが、コストも上がります。要件とコストのバランスでDR戦略を選ぶ、という判断が問われます。

4つのDR戦略（AWS公式の分類）

左ほど低コスト・遅い復旧、右ほど高コスト・速い復旧です。RTO/RPOの厳しさとコストのトレードオフで選びます。

戦略概要RTO/RPOコスト
バックアップ＆リストアデータを別リージョンにバックアップ。被災時に復元・再構築時間〜（最も遅い）最小
パイロットライトDB等の中核だけを最小構成で常時稼働させ、残りは被災時に起動・拡張数十分〜数時間低〜中
ウォームスタンバイ全構成を縮小規模で常時稼働。被災時にスケールアップして本番切替分単位中〜高
マルチサイト（アクティブ/アクティブ）複数リージョンでフル稼働。被災時もほぼ無停止ほぼゼロ最大
判断のコツ

設問はたいてい「RTO/RPOの要件」と「コスト制約」をセットで示します。「数分のRTO/RPOだがコストは抑えたい」→ ウォームスタンバイ、「ダウンタイムゼロが絶対」→ マルチサイト、「復旧に多少時間がかかってよく、とにかく安く」→ バックアップ＆リストア、というように両軸で絞り込みます。

AWS Backup による一元管理

  • AWS Backupは、EBS・RDS・Aurora・DynamoDB・EFS・S3・Storage Gateway など多数のサービスのバックアップを一元的にポリシー管理できるマネージドサービス
  • バックアッププランでスケジュール・保持期間・別リージョン/別アカウントへのコピーを定義できる（DR要件に直結）
  • Backup Vault Lockでバックアップの改ざん・早期削除を防ぎ（WORM）、ランサムウェアやコンプライアンス要件に対応

データ耐久性を支える機能（まとめ）

  • S3 バージョニング＋ライフサイクル: 誤削除からの保護と、古いバージョンの低コスト移行・失効
  • S3 クロスリージョンレプリケーション（CRR）: 地理的に離れたリージョンへ自動複製
  • RDS 自動バックアップ／スナップショット: PITR（ポイントインタイムリカバリ）と手動スナップショット
  • DynamoDB PITR: 直近35日の任意時点へ復元

章末問題

4択・全5問。まず自分で選んでから「解答と解説」を開いてください。

問1. あるWebアプリのRDS（MySQL）で、参照系クエリの急増によりレスポンスが悪化している。書き込み負荷は問題ない。可用性は現状維持で構わない。最も適切な対策はどれか。

  • A. RDSをMulti-AZ配置に変更する
  • B. リードレプリカを追加し、読み取りクエリを振り分ける
  • C. インスタンスを毎晩再起動する
  • D. スタンバイインスタンスに読み取りを向ける
解答と解説を見る

正解 B

読み取りのスケールにはリードレプリカが適切です。Multi-AZ（A）は可用性のための機能で、スタンバイは読み取りに使えず性能問題は解決しません（Dも同じ理由で誤り）。再起動（C）は無関係です。「読み取り性能＝リードレプリカ／可用性＝Multi-AZ」の区別が要点です。

問2. 注文処理システムで、フロントの受付は高速に応答しつつ、バックエンドの在庫処理が急なアクセス増でも取りこぼさないようにしたい。コンポーネントを疎結合にする最適な構成はどれか。

  • A. フロントからバックエンドを同期的に直接呼び出す
  • B. フロントとバックエンドの間にSQSキューを挟み、消費側をAuto Scalingする
  • C. バックエンドのインスタンスを1台の高性能機に置き換える
  • D. Route 53のフェイルオーバールーティングを設定する
解答と解説を見る

正解 B

SQSを挟むと、フロントはメッセージを投入して即応答でき、スパイクはキューが吸収します。消費側はキューの深さに応じてAuto Scalingし、取りこぼしを防げます。直接同期呼び出し（A）は片方の遅延・障害が波及します。垂直スケール1台（C）は単一障害点かつ上限があります。フェイルオーバー（D）は疎結合やバッファリングとは別の目的です。

問3. 複数AZに分散配置したEC2群から、同一のファイル群を同時に読み書きする共有ストレージが必要。可用性を保ちつつ実現するのに最適なサービスはどれか。

  • A. 1つのEBSボリュームを全インスタンスにアタッチする
  • B. Amazon EFS を各AZのインスタンスからマウントする
  • C. 各インスタンスのインスタンスストアを使う
  • D. インスタンスごとに独立したEBSを持たせる
解答と解説を見る

正解 B

EFSは複数AZに分散する共有ファイルシステムで、複数AZのEC2から同時マウントできます。EBSは単一AZに紐づくブロックストレージで、AZをまたぐ共有には向きません（A・D）。インスタンスストア（C）は揮発性でインスタンス停止時に消えます。「複数AZ共有＝EFS」が定石です。

問4. ある企業のDR要件は「RTO・RPOともに数分以内。ただし平常時のコストはできるだけ抑えたい」。別リージョンでのDR戦略として最も適切なものはどれか。

  • A. バックアップ＆リストア
  • B. パイロットライト
  • C. ウォームスタンバイ
  • D. マルチサイト（アクティブ/アクティブ）でフル稼働
解答と解説を見る

正解 C

ウォームスタンバイは全構成を縮小規模で常時稼働させ、被災時にスケールアップして切り替えるため、分単位のRTO/RPOを、マルチサイトより低いコストで実現できます。バックアップ＆リストア（A）とパイロットライト（B）は復旧に時間がかかり数分のRTOを満たしにくい。マルチサイト（D）は要件を満たしますが「コストを抑えたい」条件に反し、過剰です。RTO/RPOとコストの両軸で選ぶのがポイントです。

問5. EBS・RDS・DynamoDB・EFSなど複数サービスのバックアップを、統一されたスケジュールと保持ポリシーで管理し、さらに別リージョンへコピーしてDRに備えたい。最も適したサービスはどれか。

  • A. 各サービスで個別にスクリプトを組み、cronで実行する
  • B. AWS Backup のバックアッププランで一元管理し、クロスリージョンコピーを設定する
  • C. S3バージョニングだけで対応する
  • D. EC2のAMIを手動で定期取得する

正解 B

AWS Backupは多数のAWSサービスのバックアップを一元的にポリシー管理し、スケジュール・保持期間・別リージョン/別アカウントへのコピーを宣言的に構成できます。個別スクリプト（A）は運用負荷と抜け漏れリスクが高い。S3バージョニング（C）はS3のみが対象。手動AMI取得（D）は網羅性も自動化も不足します。
