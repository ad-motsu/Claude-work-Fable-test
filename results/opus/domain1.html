1-0　この章の位置づけと責任共有モデル

SAA-C03試験は4つのコンテンツドメインで構成され、第1分野「セキュアなアーキテクチャの設計」は 採点対象コンテンツの30%を占める最大の分野です。ソリューションアーキテクトには、 「動くシステム」ではなく「安全に動くシステム」を設計する力が求められます。本章は公式試験ガイドの タスクステートメント1.1（セキュアなアクセス）／1.2（セキュアなワークロード）／1.3（データセキュリティ）に 沿って構成しています。

責任共有モデル（Shared Responsibility Model）

AWSのセキュリティは、AWSと利用者の間で責任を分担する責任共有モデルが土台です。 端的に言うと、AWSは「クラウドのセキュリティ」、利用者は「クラウド内のセキュリティ」に責任を持ちます。

区分責任者主な対象例
クラウドのセキュリティAWSデータセンターの物理セキュリティ、ハードウェア、ネットワーク基盤、ハイパーバイザー、マネージドサービスの基盤ソフトウェア
クラウド内のセキュリティ利用者データの暗号化、IAMの設定、セキュリティグループ/NACL、ゲストOSのパッチ、アプリケーションのコード、ネットワーク構成

重要なのは、責任の境界がサービスの抽象度によって動く点です。EC2（IaaS）ではOSのパッチ適用は利用者の責任ですが、 RDSやLambda、S3のようなマネージドサービスではOSやランタイムの保守はAWS側に移ります。ただし、 データそのものとアクセス制御（IAM）は常に利用者の責任である、と覚えてください。

試験ポイント

「〇〇のパッチは誰の責任か」「マネージドサービスに変えると責任はどう変わるか」という問いが頻出です。 データ暗号化とIAMは常に利用者側、物理・基盤は常にAWS側、その中間（OS/ミドルウェア）はサービス種別で動く、と整理しておきましょう。

タスクステートメント 1.1｜AWSリソースへのセキュアなアクセスを設計する1-1　IAMとアクセス管理

ルートユーザーの保護

AWSアカウント作成時のメールアドレスでサインインするルートユーザーは、アカウント内のすべてに対する 無制限の権限を持ちます。日常運用では使わず、次の保護を徹底します。

  • ルートユーザーにMFA（多要素認証）を設定する。
  • ルートユーザーのアクセスキーは作成しない（作成済みなら削除）。
  • 日常の操作は、必要な権限だけを持つIAMユーザー/ロールで行う。

ルートユーザーでしか実行できない操作（アカウント設定の変更、解約、特定の課金設定など）に限って使用します。

IAMの3つのアイデンティティ：ユーザー・グループ・ロール

要素特徴認証情報
IAMユーザー人やアプリに紐づく永続的なIDパスワード／アクセスキー（長期認証情報）
IAMグループユーザーの集合。権限をまとめて付与する入れ物（グループ自体はサインイン不可）なし
IAMロール「引き受けて（Assume）」使う、権限のセット。EC2・Lambda・他アカウント・外部IdPが引き受けるSTSが発行する一時認証情報
設計の基本

アプリやAWSサービス（EC2上のアプリなど）にAWS権限を与えるときは、アクセスキーを埋め込まず、IAMロールを使うのが鉄則です。 ロールが発行する一時認証情報は自動でローテーションされ、キー漏洩リスクを大幅に下げられます。

IAMポリシーと評価ロジック

権限はポリシー（JSONドキュメント）で定義します。主な要素は Effect（Allow/Deny）、Action（許可する操作）、Resource（対象）、 Condition（条件）です。ポリシーには大きく2種類あります。

  • アイデンティティベースポリシー：ユーザー/グループ/ロールにアタッチする（「この人は何ができるか」）。
  • リソースベースポリシー：リソース側にアタッチする（「このリソースに誰が触れるか」）。S3バケットポリシーやKMSキーポリシーが代表例で、クロスアカウントアクセスの許可に使えます。

複数のポリシーが絡んだときの評価は、次の原則で決まります。

  1. デフォルトは暗黙的な拒否（Implicit Deny）。どこにも許可がなければアクセスできない。
  2. いずれかのポリシーに明示的な許可（Allow）があれば許可される。
  3. ただし、いずれかに明示的な拒否（Explicit Deny）があれば、それが最優先で、他のAllowをすべて上書きして拒否する。

さらに組織環境では、SCP（サービスコントロールポリシー）や権限境界による 「許可の上限（ガードレール）」も評価されます。SCP等で許可されていない操作は、たとえIAMポリシーでAllowでも実行できません。

試験ポイント

「IAMポリシーではAllowなのにアクセスできない」→ 明示的Denyがあるか、SCP/権限境界の上限に引っかかっているかを疑います。 評価は「明示的Deny ＞ 明示的Allow ＞ 暗黙的Deny」の順で、Denyが常に勝つと覚えましょう。

STS・ロールスイッチ・クロスアカウント

AWS STS（Security Token Service）は一時的な認証情報を発行するサービスです。 AssumeRoleでロールを引き受けると、期限付きのキーが得られます。これにより、

  • ロールスイッチ：1人のユーザーが用途に応じて権限セットを切り替える。
  • クロスアカウントアクセス：アカウントAのユーザーが、アカウントBのロールを引き受けてBのリソースを操作する。長期キーを共有せずに実現できる。

フェデレーションと IAM Identity Center

社内の既存ID基盤（Active DirectoryやOkta等の外部IdP）のアカウントでAWSにサインインさせるのがフェデレーションです。 AWSユーザーを個別に作らずに済み、退職者の管理も一元化できます。

AWS IAM Identity Center（旧 AWS Single Sign-On）は、複数のAWSアカウントや業務アプリへの シングルサインオンを提供します。外部IdPと連携し、権限セットを通じて「どのアカウントで何ができるか」を 集中管理できます。複数アカウント運用では、各アカウントにIAMユーザーを作るのではなく、Identity Centerで一元化するのが定石です。

マルチアカウントのガバナンス：Organizations・SCP・Control Tower

  • AWS Organizations：複数アカウントを組織単位（OU）で束ね、一括請求とポリシー適用を行う。
  • SCP（サービスコントロールポリシー）：OUやアカウントに適用する許可の上限。権限を付与するものではなく、 「この組織では使えるサービス/操作の枠」を定めるガードレール。ルートユーザーにも効く。
  • AWS Control Tower：ベストプラクティスに沿ったマルチアカウント環境（ランディングゾーン）を自動構築し、 ガードレールで統制する上位サービス。
よくある誤解

SCPは許可を与えません。SCPで許可されていても、IAMポリシー側でAllowしなければアクセスできません。 「SCP（上限）とIAMポリシー（実際の付与）の両方でAllow」が必要、と理解してください。

タスクステートメント 1.2｜セキュアなワークロードとアプリケーションを設計する1-2　ネットワークセキュリティとワークロード保護

VPCとサブネット設計の基本

VPC（Virtual Private Cloud）は、AWS上に構築する論理的に隔離された自分専用のネットワークです。 その中をサブネットに分割し、ルートテーブルで通信経路を制御します。セキュリティ設計の出発点は、 「インターネットに直接公開してよいものだけをパブリックに置く」という切り分けです。

種類特徴置くもの（例）
パブリックサブネットルートテーブルにインターネットゲートウェイ（IGW）へのルートがあるロードバランサー、踏み台、NATゲートウェイ
プライベートサブネットIGWへの直接ルートがない。外向き通信はNATゲートウェイ経由アプリサーバー、DB（RDS）

NATゲートウェイは、プライベートサブネットのリソースが「外へ出る（アウトバウンド）」通信を可能にしつつ、 外部から中への接続を遮断します。DBやアプリサーバーはプライベートに置き、必要な入口だけを絞るのが基本形です。

セキュリティグループ vs ネットワークACL

この2つの違いは試験頻出です。ステートフル/ステートレスの区別を確実に押さえましょう。

観点セキュリティグループ（SG）ネットワークACL（NACL）
適用単位ENI（インスタンス）レベルサブネットレベル
状態ステートフル：許可した通信の戻りは自動的に許可ステートレス：戻りの通信も明示的にルールが必要
ルール許可（Allow）のみ。拒否ルールは書けない許可と拒否（Deny）の両方が書ける
評価すべてのルールを総合して判定ルール番号の小さい順に評価し、最初に一致したもので決定
デフォルトインバウンド全拒否／アウトバウンド全許可デフォルトNACLは全許可
試験ポイント

「特定IPを明示的にブロックしたい」→ SGでは不可能なのでNACLを使う。 通常のアクセス制御はSG（ステートフルで扱いやすい）、サブネット全体の粗いガードやIP拒否はNACL、という役割分担で覚えます。

VPCエンドポイント：インターネットを経由しないアクセス

プライベートサブネットのリソースからS3やDynamoDBなどのAWSサービスへアクセスする際、 NATゲートウェイ経由でインターネットに出さずに接続できるのがVPCエンドポイントです。

  • ゲートウェイ型エンドポイント：S3・DynamoDB専用。ルートテーブルにエントリを追加する方式。追加料金なし。
  • インターフェース型エンドポイント（AWS PrivateLink）：多くのサービスに対応。サブネット内にENI（プライベートIP）を作り、そこ経由で接続する。

境界防御：WAF と Shield

サービス守る層役割
AWS WAFレイヤー7（HTTP/HTTPS）SQLインジェクションやクロスサイトスクリプティング（XSS）など、Webアプリへの攻撃をルールでブロック。CloudFront・ALB・API Gateway・AppSyncに適用
AWS Shield Standardレイヤー3/4一般的なDDoS攻撃から自動保護。全AWS利用者に無料で適用
AWS Shield AdvancedL3/4＋L7支援有料。高度なDDoS検知、DDoS対応チーム（DRT/SRT）支援、攻撃起因のスケール費用に対するコスト保護
使い分け

SQLインジェクションのようなアプリ層攻撃 → WAF、大量トラフィックによるDDoS → Shield。 「WebアプリをL7攻撃から守る」ならWAF、と即答できるようにしておきましょう。

脅威検知：GuardDuty・Macie・Inspector

  • Amazon GuardDuty：脅威検知サービス。CloudTrail・VPCフローログ・DNSログなどを機械学習で分析し、 不審なアクティビティ（不正アクセス、マルウェア通信など）を検出。エージェント不要。
  • Amazon Macie：S3内の機密データ（個人情報などのPII）を機械学習で発見・分類し、公開設定の問題も可視化する。
  • Amazon Inspector：EC2・コンテナイメージ・Lambdaの脆弱性（CVE）を自動スキャンする。
試験ポイント

キーワードで即断：「脅威・不審な挙動の検知＝GuardDuty」「S3の機密データ発見・分類＝Macie」「脆弱性スキャン＝Inspector」。役割が混同されやすいので対応表で暗記します。

アプリ認証とハイブリッド接続

  • Amazon Cognito：Web/モバイルアプリのユーザー認証（ユーザープール）と、認証済みユーザーへのAWSリソースアクセス付与（IDプール）を提供。
  • AWS Site-to-Site VPN：インターネット経由でオンプレとVPCをIPsecで接続。短期間・低コストで開始できる。
  • AWS Direct Connect：AWSへの専用線接続。安定した帯域と低レイテンシ、通信料の削減。インターネットを経由しない。

タスクステートメント 1.3｜適切なデータセキュリティ管理を判断する1-3　データ保護と暗号化

AWS KMS と暗号化の考え方

AWS KMS（Key Management Service）は暗号鍵を集中管理するサービスです。多くのAWSサービスが 「保管中データの暗号化」でKMSを利用します。KMSの中核概念を押さえましょう。

  • エンベロープ暗号化：実データは高速なデータキーで暗号化し、そのデータキー自体をKMSキー（鍵暗号鍵）で暗号化する二段構え。KMSキー本体はKMSの外に出ません。
  • KMSキーの種類：
    • カスタマー管理キー：利用者が作成・管理。キーポリシー、ローテーション設定、アクセス制御を細かく制御できる。
    • AWS管理キー（例 aws/s3）：サービスが自動作成・管理。設定はできないが手軽。
    • AWS所有キー：AWSが所有し利用者からは見えない。
  • キーポリシー：KMSキーへのアクセスを定めるリソースベースポリシー。鍵の利用可否は最終的にここで制御する。

鍵のローテーション（2026年時点の仕様）

鍵の種類自動ローテーション周期
カスタマー管理キー任意（デフォルト無効）。有効化できるデフォルト365日。カスタム周期も指定可。加えてオンデマンドローテーションも可能
AWS管理キー常に有効（無効化不可）毎年（約365日）。2022年5月に3年周期から1年周期へ変更された
AWS所有キーサービスが管理サービス側の戦略に従う
補足

自動ローテーションで新しい鍵マテリアルが作られても、古い鍵マテリアルは保持されます（KMS生成鍵はキー削除時まで保持）。 過去に暗号化したデータも復号できるため、アプリの変更は不要です。ローテーションされてもキーID（論理リソース）は変わりません。

S3の暗号化（保管中）

Amazon S3は2023年1月5日以降、すべての新規オブジェクトを既定で暗号化します。 ベースラインはSSE-S3（Amazon S3管理キー・AES-256）で、追加料金なし・性能影響なしで自動適用され、 新規アップロードの暗号化は無効化できません。要件に応じて、より強い管理方式を選べます。

方式鍵の管理使いどころ
SSE-S3S3が管理（AES-256）既定。手間なく暗号化したい
SSE-KMSKMSキー鍵の使用をCloudTrailで監査し、アクセスをキーポリシーで細かく制御したい
DSSE-KMSKMSキーで二重暗号化規制等で多層の暗号化が求められる
SSE-C顧客が鍵を提供鍵をAWSに預けず自分で管理したい

転送中はHTTPS（TLS）でアクセスし、バケットポリシーで暗号化通信を強制できます。

EBSの暗号化

  • EBS暗号化はKMSキーを用い、保管中データと、インスタンス〜ボリューム間の転送中データの両方を保護する。
  • デフォルト暗号化はアカウント×リージョン単位の設定。有効化すると、以後作成するボリュームが自動で暗号化される（既定の鍵は指定しなければ aws/ebs）。
  • 既存の非暗号化ボリュームをその場で暗号化に変えることはできない。スナップショットを取り、その暗号化コピーから新しいボリュームを作る、という手順を踏む。
  • 暗号化ボリュームから作ったスナップショット、さらにそこから復元したボリュームは常に暗号化される（暗号化は解除できない）。

転送中の暗号化とACM

通信経路の保護（転送中データの暗号化）はTLSで行います。 AWS Certificate Manager（ACM）はTLS証明書の発行・管理・自動更新を担い、 ELB・CloudFront・API Gatewayなどに統合できます。ACMが発行するパブリック証明書は自動更新されるため、 証明書失効による障害を防げます。

Secrets Manager と認証情報の保護

  • AWS Secrets Manager：DBパスワードやAPIキーなどのシークレットを暗号化保管し、自動ローテーションもできる。アプリはコードに秘密情報を埋め込まず、実行時に取得する。
  • Systems Manager Parameter Store：設定値やシークレットを保管。標準パラメータは無料。自動ローテーションが要る、あるいは高度なシークレット管理なら Secrets Manager が適する。
試験ポイント

「DB認証情報を安全に保管し定期的に自動ローテーションしたい」→ Secrets Manager。 「保管中データを暗号化し鍵の使用を監査したい」→ SSE-KMS＋カスタマー管理キー。 「TLS証明書を自動更新したい」→ ACM。この対応で多くの問題に答えられます。

理解度確認問題（5問）

各問、選択肢を選んでから「解答と解説を見る」を開いてください。実際の試験と同じ、シナリオ形式の4択です。

問1
アカウントAで稼働するLambda関数から、アカウントBのS3バケットにあるオブジェクトを読み取る必要がある。長期的な認証情報を共有せずに実現する、最もセキュアな方法はどれか。
  • A. アカウントBにIAMユーザーを作成し、そのアクセスキーをアカウントAのLambdaの環境変数に設定する
  • B. アカウントBにクロスアカウント用のIAMロールを作成し、バケットポリシーで許可、アカウントAのLambda実行ロールにそのロールをAssumeRoleさせる
  • C. S3バケットを一時的にパブリック公開にして読み取る
  • D. アカウントBのルートユーザーのアクセスキーをLambdaに設定する
解答と解説を見る

正解：B

クロスアカウントアクセスは、相手アカウントにIAMロールを用意し、STSのAssumeRoleで一時認証情報を得るのが定石です。 リソースベースポリシー（バケットポリシー）と組み合わせて最小権限で許可できます。

A・Dは長期のアクセスキーを共有しており、要件（共有しない）に反し漏洩リスクも高い。特にDのルートキー利用は重大なアンチパターン。 Cはデータを世界に公開してしまい論外です。

問2
あるIAMユーザーのアイデンティティベースポリシーには特定のS3操作を許可するAllowが付いている。しかし操作するとアクセス拒否になる。組織ではそのユーザーが属するOUに、当該S3操作を許可していないSCPが適用されていた。この状況の説明として正しいものはどれか。
  • A. IAMポリシーでAllowされていれば、SCPの内容に関わらず操作できるはずである
  • B. SCPは許可の上限（ガードレール）であり、SCPで許可されていない操作はIAMのAllowがあっても実行できない
  • C. SCPは請求専用の機能で、権限には影響しない
  • D. ルートユーザーで操作すればSCPを回避できる
解答と解説を見る

正解：B

SCPは許可の上限を定めるもので、実際にアクセスするには「SCP（上限）」と「IAMポリシー（付与）」の両方でAllowが必要です。 SCPで許可されていなければ、IAM側がAllowでも拒否されます。

Aは評価順序の誤解。Cは誤り（SCPは権限のガードレール）。DもSCPはルートユーザーにも効くため回避できません。

問3
医療系企業が、S3に保管するデータを暗号化したうえで、「どの鍵がいつ誰の操作で使われたか」を監査ログで追跡し、鍵へのアクセスをきめ細かく制御したいと考えている。最も適した暗号化方式はどれか。
  • A. SSE-S3（Amazon S3管理キー）
  • B. SSE-KMS（カスタマー管理キー）
  • C. 暗号化は行わず、バケットを非公開にするだけにする
  • D. クライアント側で各自バラバラに暗号化し、鍵管理はしない
解答と解説を見る

正解：B

鍵の使用をCloudTrailで監査し、キーポリシーでアクセスを細かく制御できるのはSSE-KMSです。 カスタマー管理キーなら、ローテーションやアクセスポリシーも自分で管理できます。

Aは手軽ですが、鍵利用の監査やアクセス制御の粒度はKMSに劣ります。Cは暗号化要件を満たしません （なお2023年以降、新規オブジェクトは最低でもSSE-S3で暗号化されます）。Dは鍵管理が破綻し運用に耐えません。

問4
公開Webアプリケーションが、SQLインジェクションを狙ったリクエストを受けている。アプリケーション層（レイヤー7）でこうした不正リクエストをフィルタリングしてブロックする、最も適切なサービスはどれか。
  • A. AWS Shield Standard
  • B. ネットワークACL（NACL）
  • C. AWS WAF
  • D. セキュリティグループ
解答と解説を見る

正解：C

SQLインジェクションやXSSといったL7のWeb攻撃をルールでブロックするのはAWS WAFです。 CloudFrontやALB、API Gatewayに適用します。

AのShieldはL3/4のDDoS対策が主目的。B・DはIP/ポート単位の制御であり、HTTPリクエストの中身（SQL構文など）を検査できません。

問5
プライベートサブネットのEC2インスタンスから、同一リージョンのS3バケットへアクセスさせたい。通信をインターネットに一切出さず、かつ余計なコストを避けたい。最適な構成はどれか。
  • A. インターネットゲートウェイ（IGW）を追加してパブリック経路で接続する
  • B. NATゲートウェイを設置し、インターネット経由でS3にアクセスする
  • C. S3向けのゲートウェイ型VPCエンドポイントを作成し、ルートテーブルに追加する
  • D. インスタンスをパブリックサブネットに移動し、パブリックIPを付与する
解答と解説を見る

正解：C

S3・DynamoDBにはゲートウェイ型VPCエンドポイントがあり、ルートテーブルにエントリを追加するだけで インターネットを経由せずプライベートに接続できます。追加料金もかかりません。

A・Dはインスタンスをインターネットに露出させます。BのNATゲートウェイはインターネット経由であり、要件（外に出さない）に反するうえ時間・処理料金が発生します。
