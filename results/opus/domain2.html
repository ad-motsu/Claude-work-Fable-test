2-0　この章の位置づけ：弾力性とは何か

第2分野「弾力性に優れたアーキテクチャの設計」は採点対象コンテンツの26%を占め、第1分野に次ぐ大分野です。 ここでいう「弾力性（resilience）」は、大きく2つの能力に分解できます。

  • スケーラビリティ（伸縮性）：負荷の増減に応じて、処理能力を自動で増減できること。過剰投資も過負荷も避ける。
  • 可用性・フォールトトレランス（耐障害性）：一部のコンポーネントやAZが壊れても、システム全体としては動き続けること。

この2つを支える物理的な土台がアベイラビリティーゾーン（AZ）とリージョンです。1つのリージョンは 互いに物理的に分離された複数のAZで構成され、各AZは独立した電源・冷却・ネットワークを持ちます。 つまりAZは「独立した障害の単位」であり、複数AZにまたがって配置することが高可用性設計の出発点になります。

設計思想

AWSの信頼性設計の合言葉は「Design for failure（障害を前提に設計する）」です。 「壊れないコンポーネント」を目指すのではなく、「壊れても影響が波及しない」「壊れたら自動で置き換わる」構造を作ります。 本章の技術は、すべてこの思想の具体化だと捉えてください。

タスクステートメント 2.1｜スケーラブルで疎結合なアーキテクチャを設計する2-1　スケーラブルで疎結合なアーキテクチャ

スケールアップ vs スケールアウト

方式内容特徴
垂直スケーリング（スケールアップ）インスタンスをより大きなタイプに変更する単純だが上限があり、変更時に停止を伴うことが多い。単一障害点になりやすい
水平スケーリング（スケールアウト）同じ役割のインスタンスを台数で増やす理論上ほぼ無限に伸ばせ、複数台なので可用性も上がる。クラウドの基本戦略

SAAでは、原則として水平スケーリングを優先します。水平に増やせる設計＝ステートレス設計（後述）とセットで考えます。

Elastic Load Balancing（ELB）

複数のインスタンスに負荷を分散し、異常なターゲットを切り離す入口がELBです。3種類の使い分けが頻出です。

種類レイヤー主な用途
Application Load Balancer（ALB）L7（HTTP/HTTPS）Web/HTTPアプリ。パスやホスト名によるルーティング、コンテナ（ECS）、Lambda、WebSocketに対応
Network Load Balancer（NLB）L4（TCP/UDP/TLS）超低レイテンシ・高スループット。静的IP/Elastic IPが必要な場合や、極端な性能要件
Gateway Load Balancer（GWLB）L3/L4サードパーティ製の仮想アプライアンス（ファイアウォール、IDS/IPS）を透過的に挿入
試験ポイント

「HTTPのパスで振り分けたい／コンテナに分散」→ ALB。「TCP/UDPで極限の性能・固定IPが要る」→ NLB。 「セキュリティ製品を挟みたい」→ GWLB。この3択を即答できるように。

EC2 Auto Scaling

EC2 Auto Scaling グループ（ASG）は、指定した希望台数（desired capacity）を維持し、 需要に応じて台数を増減させます。ELBと組み合わせるのが定番です。

  • 異常なインスタンスをヘルスチェックで検知し、自動で置き換える（自己修復）。
  • 複数AZに分散して起動し、AZ障害の影響を緩和する。
  • スケーリングポリシー：ターゲット追跡（例：平均CPU 50%を維持）、ステップ/シンプル、スケジュール（予測可能な波に対応）。

疎結合：SQS・SNS・EventBridge

コンポーネント同士を直接つなぐと、片方の障害や急な負荷がもう片方に波及します。間にマネージドな緩衝材を挟んで 疎結合（loose coupling）にするのが弾力性設計の核心です。

サービスモデル役割
Amazon SQSキュー（ポーリング型）メッセージを一時的に溜め、受信側が自分のペースで処理。負荷の急増を吸収し、送信側と受信側を切り離す
Amazon SNSパブリッシュ/サブスクライブ（プッシュ型）1つのメッセージを複数の購読先へ同報（ファンアウト）。SQS・Lambda・HTTP・Emailなどへ配信
Amazon EventBridgeイベントバスAWSサービスやSaaS、独自アプリのイベントをルールで振り分け。イベント駆動アーキテクチャの中枢

SQSの2タイプ

  • 標準キュー：ほぼ無制限のスループット、少なくとも1回配信、順序はベストエフォート。
  • FIFOキュー：厳密な順序と重複排除（正確に1回）。スループットには上限がある。
代表パターン：SNS + SQS ファンアウト

SNSトピックに複数のSQSキューをサブスクライブしておくと、1件のイベントを複数の処理系へ確実に配布しつつ、 各処理系はキューで負荷を平準化できます。「1つの出来事を複数チームが別々に処理」という要件の定番解です。

ワークフローとサーバーレス

  • AWS Step Functions：複数の処理（Lambdaや各種サービス）をステートマシンとして順序・分岐・リトライ込みで オーケストレーションする。長時間・多段の業務フローの調停役。
  • AWS Lambda：イベントに応じて関数を実行するサーバーレスコンピュート。サーバー管理不要で自動スケール。
  • AWS Fargate：ECS/EKS向けのサーバーレスコンテナ実行環境。EC2を管理せずコンテナを動かす。

ステートレス設計・多層・キャッシュ

  • ステートレス設計：セッション状態をインスタンス内に持たず、DynamoDBやElastiCache（外部）に保存する。 どのインスタンスでも同じリクエストを処理でき、水平スケーリングとインスタンス入れ替えが安全に行える。
  • 多層アーキテクチャ：Web層／アプリ層／データ層を分離し、各層を独立にスケール・保護する。
  • キャッシュ：Amazon CloudFront（エッジでコンテンツをキャッシュしオリジン負荷とレイテンシを削減）、 Amazon ElastiCache（DB前段のインメモリキャッシュで読み取りを高速化）。

  • Amazon EC2 Auto Scaling: what-is-amazon-ec2-auto-scaling.html
  • Amazon SQS: SQS Developer Guide ／ SNS: sns/dg/welcome.html
  • Amazon EventBridge: eb-what-is.html ／ Step Functions: step-functions/dg/welcome.html

タスクステートメント 2.2｜高可用性、フォールトトレラントなアーキテクチャを設計する2-2　高可用性・フォールトトレラント設計

マルチAZが高可用性の基本

単一障害点（SPOF）を排除する最初の一手は、コンポーネントを2つ以上のAZに分散することです。 ELB＋Auto Scalingで複数AZにインスタンスを配置し、DBはマネージドなマルチAZ構成にする——これが定番の可用性テンプレートです。

Route 53：DNSによるトラフィック制御とフェイルオーバー

Amazon Route 53はDNSサービスで、ヘルスチェックと組み合わせて障害時の切り替えを実現します。 主なルーティングポリシー：

ポリシー用途
シンプル単一リソースへ
フェイルオーバープライマリが異常ならセカンダリへ切り替え（アクティブ/パッシブ）
加重（Weighted）割合を指定して分配（カナリアリリース等）
レイテンシー最も低遅延のリージョンへ
位置情報 / 地理的近接ユーザーの地域に応じて振り分け
複数値回答複数の正常なエンドポイントを返す

データベースの可用性

RDS：マルチAZ vs リードレプリカ

機能目的仕組み
マルチAZ配置可用性（災害復旧ではなく高可用性）別AZに同期スタンバイを保持し、障害時に自動フェイルオーバー（DNSの向き先を切替）。標準のスタンバイは読み取りには使わない
リードレプリカ読み取りのスケール非同期レプリケーション。参照負荷を分散。別リージョンにも作成可。昇格は手動で、DRにも応用できる
混同注意

「AZ障害時に自動で切り替わる高可用性」＝マルチAZ。「読み取り性能を上げたい」＝リードレプリカ。 リードレプリカは自動フェイルオーバーの仕組みではありません（昇格は手動）。ここは頻出の引っかけです。

Aurora

  • ストレージは3つのAZにまたがり6つのコピーで自動複製され、高い耐久性と可用性を持つ。
  • 最大15個のリードレプリカ、高速な自動フェイルオーバー。
  • Aurora グローバルデータベース：リージョン間を通常1秒未満のレイテンシで複製し、セカンダリを1分未満で昇格可能。クロスリージョンDRに強い。

DynamoDB

  • フルマネージドで、標準で複数AZに複製され高可用。
  • グローバルテーブル：複数リージョンで読み書きできるアクティブ/アクティブ構成（競合は「最後の書き込み優先」で解決）。

ストレージの耐久性

  • Amazon S3：オブジェクトを複数AZに冗長格納し、99.999999999%（イレブンナイン）の耐久性を提供。
  • Amazon EFS：リージョン内の複数AZに冗長化された共有ファイルシステム。複数のインスタンスから同時マウント可能。

接続の安定化：RDS Proxy

Amazon RDS ProxyはDB接続をプールして再利用し、大量の同時接続（コネクションストーム）からDBを守ります。 フェイルオーバー時の切り替え時間短縮にも寄与し、Lambdaのような大量接続を生むワークロードと相性が良いです。

試験ポイント：単一障害点の排除

「可用性を高めよ」という設問では、①複数AZ配置 ②ELB＋Auto Scaling ③マネージドなマルチAZ DB ④マネージドサービスの活用が定石。 逆に「単一インスタンス」「単一AZ」「垂直スケールのみ」はSPOFの兆候として避けます。

タスクステートメント 2.2｜データの耐久性・可用性の確保と適切なDR戦略の選択2-3　バックアップとDR戦略

RTO と RPO

DR（災害対策）の要件は、必ず次の2つの指標で語られます。

指標意味問い
RTO（目標復旧時間）障害発生から復旧までに許容できる時間「どれだけ止まってよいか」
RPO（目標復旧時点）障害時に失ってよいデータの量（直近バックアップまでの時間幅）「どこまでのデータ喪失を許容できるか」

RTO・RPOを小さく（＝ダウンタイムとデータ喪失を少なく）するほど、一般にコストと複雑さは増します。このトレードオフで戦略を選びます。

4つのDR戦略

AWSのDR戦略は、コスト・RTO/RPOの小ささの順に4段階で整理されます（下にいくほど高速・高コスト）。

戦略DRリージョンの状態RTO / RPOコスト
バックアップ＆リストアバックアップのみ。障害時にインフラを再デプロイしデータを復元最も長い（数時間）最も低い
パイロットライトデータは常時レプリケーション、コアのDB/ストレージは常時稼働。アプリサーバーは「電源オフ」で待機し、障害時に起動・スケール短い（数十分）低〜中
ウォームスタンバイ縮小版だが完全に稼働中の環境が常時トラフィックを（低容量で）処理可能。障害時はスケールアップするだけより短い（数分）中〜高
マルチサイト アクティブ/アクティブ複数リージョンで同時に本番稼働。全リージョンがトラフィックを処理ほぼゼロ最も高い
試験ポイント：パイロットライト vs ウォームスタンバイ

両者の違いは「すぐにトラフィックを捌けるか」。パイロットライトはアプリを起動してからでないと処理できない（コア基盤だけ常時稼働）。 ウォームスタンバイは最初から（縮小容量で）稼働中で、スケールアップするだけ。RTOをより短くしたいならウォームスタンバイ、コストを抑えたいならパイロットライト。

AWS Backup と S3 の保護

  • AWS Backup：EBS・EC2・RDS/Aurora・DynamoDB・EFS・Storage Gatewayなどのバックアップを一元的に設定・スケジュール・監視できる。 クロスリージョン／クロスアカウントのコピーにも対応し、DRやランサムウェア対策に有効。
  • S3 クロスリージョンレプリケーション（CRR）：オブジェクトを別リージョンへ非同期複製し、ほぼゼロに近いRPOでデータを保護。 バージョニングと併用すると、誤削除・改ざんからの復元点を確保できる。
補足

継続的レプリケーション（CRR、Aurora Global DB等）はRPOをほぼゼロにできますが、データ破損や誤削除はそのまま複製されうるため、 ポイントインタイムのバックアップと組み合わせるのが公式の推奨です。

理解度確認問題（5問）

各問、選択肢を選んでから「解答と解説を見る」を開いてください。実際の試験と同じ、シナリオ形式の4択です。

問1
Webアプリのトラフィックが時間帯で大きく変動する。可用性を保ちつつ、負荷に応じて処理能力を自動で増減させ、異常なサーバーは自動で置き換えたい。最適な構成はどれか。
  • A. 1台の大型EC2インスタンスを単一AZに配置し、負荷が上がったらより大きなタイプへ変更する
  • B. 複数AZにまたがるEC2 Auto Scalingグループを作り、ALBの背後に配置する
  • C. 固定台数のEC2を単一AZに並べ、DNSラウンドロビンで分散する
  • D. オンプレミスにサーバーを増設する
解答と解説を見る

正解：B

ALB＋複数AZのAuto Scalingグループが定番解。需要に応じた自動スケール、異常インスタンスの自動置換、AZ分散による可用性をすべて満たします。

Aは垂直スケールのみで単一AZ・単一障害点、変更時に停止も伴う。Cは固定台数で自動増減できず単一AZ。Dはクラウドの弾力性を活かせません。

問2
フロントエンドが受け付けた画像処理リクエストが、キャンペーン時に急増してバックエンドの処理サーバーを圧迫し、リクエストが失われることがある。バックエンドを過負荷から守り、送受信を切り離す最適な方法はどれか。
  • A. フロントエンドとバックエンドの間にAmazon SQSキューを挟み、処理側をキュー長に応じてAuto Scalingする
  • B. バックエンドのインスタンスを常に最大台数で起動しておく
  • C. フロントエンドからバックエンドへ同期的に直接呼び出す
  • D. RDSのインスタンスタイプを大きくする
解答と解説を見る

正解：A

SQSでリクエストをバッファリングすれば、急増分はキューに溜まり、処理側は自分のペースで消化できます。 送信側と受信側が疎結合になり、スパイクによる取りこぼしを防げます（キュー長でAuto Scalingも可能）。

Bは常時最大でコスト過大かつスパイク超過時に破綻。Cは密結合で過負荷が直接波及。Dは処理サーバーの過負荷とは無関係です。

問3
本番のRDS(MySQL)で、AZ障害が起きてもできるだけ短いダウンタイムで自動的に復旧できるようにしたい。最も適切な設定はどれか。
  • A. リードレプリカを1つ作成し、障害時に手動で昇格する
  • B. RDSをマルチAZ配置にする
  • C. 日次スナップショットを取得しておく
  • D. インスタンスタイプを大きくする
解答と解説を見る

正解：B

マルチAZ配置は別AZに同期スタンバイを持ち、AZ障害時に自動フェイルオーバーします。「自動・短時間・高可用性」の要件に合致します。

Aのリードレプリカは読み取りスケール用で非同期、昇格は手動のため要件に合いません。Cは復旧に時間がかかりRPO/RTOが大きい。Dは可用性向上とは無関係です。

問4
あるシステムのDR要件は「リージョン障害からの復旧を数十分で行いたいが、平常時のコストはできるだけ抑えたい」。DRリージョンではデータを常時複製し、コアのDBは動かしておくが、アプリサーバーは通常は停止させておく。この方針に最も合致するDR戦略はどれか。
  • A. バックアップ＆リストア
  • B. パイロットライト
  • C. マルチサイト アクティブ/アクティブ
  • D. 単一AZ構成
解答と解説を見る

正解：B

パイロットライトは、データを常時レプリケーションしコアのDB/ストレージは常時稼働させつつ、 アプリサーバーは「電源オフ」で待機させ、障害時に起動・スケールする方式。「数十分のRTO」と「低コスト」の両立に合致します。

Aはインフラ再デプロイからで復旧が最も遅い。Cはほぼゼロ復旧だが最も高コストで要件（コスト抑制）に反する。Dはそもそもリージョン障害に対応できません。

問5
Webアプリを複数のEC2にスケールアウトしたところ、ログイン状態が頻繁に切れる不具合が起きた。各インスタンスがセッション情報をローカルメモリに保持していることが原因だった。水平スケーリングを安全にする最適な対処はどれか。
  • A. インスタンスを1台に戻す
  • B. セッション情報をAmazon ElastiCacheやDynamoDBなどの外部ストアに保存し、アプリをステートレスにする
  • C. 各インスタンスのメモリを増やす
  • D. すべてのユーザーを常に同じ1台に固定する
解答と解説を見る

正解：B

セッション状態を外部（ElastiCache/DynamoDB）に外出ししてアプリをステートレスにすれば、 どのインスタンスでも同じユーザーを処理でき、水平スケーリングやインスタンス入れ替えが安全になります。

Aは可用性・拡張性を捨てる後退。Cは根本原因（状態の局所保持）を解決しない。Dは実質1台依存となりスケールアウトの利点と可用性を損ないます。
